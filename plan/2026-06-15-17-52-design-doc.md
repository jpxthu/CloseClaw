# Plan: Agent Config 结构对齐设计文档（PR 1/3）

## 来源
Design Doc 差异扫描：`docs/design/agent/agent-config.md`（F2b 深度对比发现 8 处 gap，本 plan 覆盖 gap 1-4）

## 项目路径
/home/admin/code/closeclaw

## 目的
修复 `AgentConfig` 结构与设计文档 `agent-config.md` 之间的 4 处代码偏离：
1. `agents.json` 注册清单加载路径与文档不一致
2. `AgentConfig` 包含文档未定义的 `max_child_depth` 字段（且 `check_max_depth` 为死代码）
3. `AgentConfig` 包含运行时字段 `state`/`created_at`，违反文档"纯配置值对象"声明
4. `AgentConfig` 包含文档未定义的 `communication`/`wait_timeout_secs`/`grace_period_secs` 字段

设计文档是最高准则，代码必须向文档对齐。不得修改 design doc。

## 思路
按 gap 分 4 个子步骤，每个步骤独立可验证：
- Step 1.1：修正 `agents.json` 路径（最小改动，单文件）
- Step 1.2：移除 `max_child_depth` 及相关死代码
- Step 1.3：将 `state`/`created_at` 从 `AgentConfig` 迁移到运行时层
- Step 1.4：移除未文档化字段
- Step 1.5：补充 UT 覆盖

每个步骤改动控制在 100 行以内，严格遵循 CONTRIBUTING.md 规范。

## 开发步骤

开发必读：
- **必须**阅读 CONTRIBUTING.md `CONTRIBUTING.md`
- **必须**阅读 `docs/design/agent/agent-config.md`

### Step 1.1：修正 agents.json 注册清单路径

**问题**：代码从 `~/.closeclaw/agents.json` 加载用户级注册清单，设计文档数据流章节指定路径为 `~/.closeclaw/config/agents.json`。

**改动范围**：
- `src/config/agent_loader.rs`：`self.config_dir.join("agents.json")` → `self.config_dir.join("config").join("agents.json")`
- `src/config/agent_loader_tests.rs`：更新测试中的路径
- `src/config/manager_reload_tests.rs`：更新测试中的路径

**验收标准**：
- 代码中用户级 `agents.json` 路径为 `~/.closeclaw/config/agents.json`
- 所有相关测试路径已同步更新
- `cargo build` 通过

### Step 1.2：移除 max_child_depth 及死代码

**问题**：`AgentConfig.max_child_depth`（默认 3）与设计文档仅描述的 `subagents.maxSpawnDepth`（默认 1）语义重叠。`check_max_depth` 函数使用 `max_child_depth` 但无调用方（死代码）。

**改动范围**：
- `src/agent/config/mod.rs`：移除 `max_child_depth` 字段及 `default_max_child_depth` 函数
- `src/agent/communication.rs`：移除 `check_max_depth` 函数及相关错误变体
- `src/agent/config/config_tests.rs`：移除测试中 `max_child_depth` 的使用
- 确认 `check_max_depth` 确实无调用方（grep 验证）

**验收标准**：
- `AgentConfig` 不再包含 `max_child_depth` 字段
- `check_max_depth` 函数已移除
- `cargo build` 通过
- `grep -rn "max_child_depth" src/` 无结果

### Step 1.3：迁移运行时字段 state/created_at

**问题**：设计文档声明"Agent 配置档案是纯配置值对象，不包含运行时可变字段"，但 `AgentConfig` 包含 `state`（Running/Suspended/Stopped）和 `created_at`（DateTime）。

**改动范围**：
- `src/agent/config/mod.rs`：从 `AgentConfig` 移除 `state` 和 `created_at` 字段，移除 `AgentConfigState` 枚举
- 确认 `state`/`created_at` 的消费方（grep 查找），将消费方改为从 Session 模块获取或使用默认值
- `src/agent/config/config_tests.rs`：移除测试中相关字段

**注意**：需先 grep 确认 `state` 和 `created_at` 的所有使用方，评估迁移影响。如果消费方过多（>5 处），需进一步拆分。

**验收标准**：
- `AgentConfig` 不再包含 `state`、`created_at`、`AgentConfigState`
- `cargo build` 通过
- 相关测试已更新

### Step 1.4：移除未文档化字段

**问题**：`AgentConfig` 包含 `communication`（CommunicationConfig）、`wait_timeout_secs`、`grace_period_secs`，设计文档无定义。

**改动范围**：
- `src/agent/config/mod.rs`：从 `AgentConfig` 移除这 3 个字段
- 确认消费方，将消费方改为使用默认值或移除相关逻辑
- `src/agent/communication.rs`：评估 `CommunicationConfig` 及相关函数是否仍有消费方
- `src/agent/config/config_tests.rs`：更新测试

**注意**：`communication` 字段可能有较多消费方，需先评估。如果影响过大，此步骤可能需要进一步拆分或标记为 blocked。

**验收标准**：
- `AgentConfig` 仅包含设计文档定义的字段
- `cargo build` 通过
- 相关测试已更新

### Step 1.5：补充 UT

**要求**：覆盖 Step 1.1-1.4 的所有逻辑分支和 corner case。

**改动范围**：
- `src/config/agent_loader_tests.rs`：补充路径变更的测试
- `src/agent/config/config_tests.rs`：更新因字段移除而变更的测试，补充新场景

**验收标准**：
- `cargo test` 通过
- 新增/修改的测试覆盖所有变更逻辑
