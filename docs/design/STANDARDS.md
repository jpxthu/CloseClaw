# 设计文档（docs/design/）写作标准

## 定位

从**产品设计（PM）角度**描述模块架构、职责、内部逻辑流转、与其他模块的信息传递。讲清楚**最新设计的意图和原因**，不讲历史。

## 章节结构

```
# NN 模块名或主题

## 职责
- 一句话：这个模块做什么

## 设计意图
- 为什么需要这个设计，解决了什么问题
- 当前状态下的痛点

## 核心思路
- 设计的核心出发点
- 关键机制和策略（不写代码，用自然语言描述）

## 逻辑流转
- 数据经过模块的处理流程（可用 ASCII 图）
- 关键判断点和分支条件

## 与其他模块的关系
- 上游/下游模块及传递的信息格式
- 明确标注"无关"的部分，防止误解

## 关键决策
- 3~5 个最重要的设计取舍
- 每条说清楚：为什么选 A 不选 B
```

## 写作原则

- **不写函数名、变量名、代码片段**——这是产品设计文档，不是技术规格
- **适当写实现原理和核心机制**——用自然语言讲，不列代码
- **控制长度**——单个功能模块文档不超过 6 节的篇幅。大模块（多个子功能）拆为多个文档
- **文件夹组织**——`docs/design/<模块名>/NN-xxx.md`，大模块按功能区分子目录

## 不写什么

- 历史决策、演进过程
- 代码里直接能看到的字段列表、函数签名
- 详细实现细节（放到代码注释）
- 未来计划
- 开发步骤、验收标准（那是开发 issue 的事）
