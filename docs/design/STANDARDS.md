# 产品文档写作标准

## 定位

从产品角度描述模块的职责、架构、运转方式、模块间交互。讲清楚产品**应该是什么样**，不讲历史、不讲现状问题、不讲设计取舍过程。

本标准管产品文档。API 设计、测试用例、开发记录另立文档，不混入产品文档。

## 术语

- **上游**：调用本模块的其他模块
- **下游**：本模块调用的其他模块
- **数据流**：输入 → 处理 → 输出的完整路径

## 章节结构

固定 4 节，不增不减：

```
# 模块名称

## 概述
- 一句话：这个模块做什么

## 架构
- 核心机制和运转方式，用自然语言描述
- 可配 ASCII 图描述结构关系（用等宽字符，箭头用 -> / <-，层级缩进 2 空格）
- 子功能超过 300 字或涉及独立数据模型 → 拆分为独立 `<功能名>.md`
- 子功能在 200 字以内且无独立数据流 → 合并在一个 README 中分小节

## 数据流
- 输入 → 处理 → 输出的完整路径
- 关键判断点和分支条件
- 可配 ASCII 流程图

## 模块关系
- 上游：谁调这个模块，传什么进来
- 下游：这个模块调谁，传什么出去
- 无关：调用或数据流无关联、但名称或功能容易混淆的模块，标注"（无调用关系）"
```

## 文件组织

```
docs/design/
├── README.md                 ← 文档索引（每个模块一行，含一句话简述）
├── <模块名>/
│   ├── README.md             ← 模块产品文档（4 节）
│   └── <子功能>.md           ← 子功能产品文档（4 节）
```

目录结构尽量与 `src/` 下的模块目录对应。若产品名与目录名不一致，以产品名为文件夹名。跨模块主题放在主要受益模块下。

## 红线

**不写**：
- 第 5 节或任何超出 4 节的内容
- 借鉴来源（"参考 Claude Code..."）
- 现状问题（"当前代码中..."）
- 自问自答（"为什么选 A 不选 B"）
- 编号前缀、版本号等历史标记
- 函数名、变量名、代码片段
- 未来计划、开发步骤
- 验收标准
