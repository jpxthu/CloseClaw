# 设计文档写作标准

## 定位

设计文档（`docs/design/*.md`）描述**设计决策、历史上下文、为什么这么做**。

模块规格书（`src/<模块>/SPEC.md`）描述**模块是什么——职责、接口、架构**。

两者互补，不重复。

## 设计文档写什么

- **设计动机**：为什么需要这个设计，解决了什么问题
- **历史决策**：为什么没有用方案 A/B，最终为什么选了方案 C
- **覆盖关系**：本文档替代了哪些旧文档/旧版本的哪些部分
- **实现要点**：关键设计点的实现摘要（不写完整实现细节）
- **未解决问题**：已知但暂未处理的问题

## 设计文档不写什么

- 完整实现细节（那是代码的事）
- 代码里直接能看到的字段列表、函数签名
- 未来计划（"考虑支持..."、"后续可扩展..."）
- 已经废弃的设计（废弃设计保留历史追溯价值，但明确标注废弃原因）

## 文件命名

`NN-<模块名或主题>.md`，编号与原 design doc 顺序对应。

如 `43-new-session-injection-chain.md`（对应 workspace design/43 号文档）。

## 与 SPEC.md 的关系

- 设计文档（design）回答"为什么"
- 模块规格书（SPEC）回答"是什么"

两者分离的好处：
- 规格书跟着代码走，代码改了同步改
- 设计文档独立演进，记录历史决策

## 覆盖关系标注

如果某设计文档替代了旧文档的某些部分，在文档头部标注：

```markdown
> 替代了 doc #27 的 §出站处理器详细设计 部分
> 替代了 doc #34-session-compaction.md（整体）
```

如果某设计文档被新文档替代，在索引（README.md）中标注 `⚠️ 部分废弃` 或 `❌ 废弃`。

## AI 可读性

设计文档面向人类开发者，结构化程度低于 SPEC。
如需 AI 可读规格，请查阅对应模块的 `SPEC.md`。
