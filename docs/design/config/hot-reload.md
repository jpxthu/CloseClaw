# hot-reload

## 概述

配置热重载使 CloseClaw 在运行时修改配置文件后自动生效，无需重启 daemon 或中断已有会话。监听配置目录文件变更，增量重载变更文件，校验通过后推送到内存和已存在会话。

## 架构

### 核心机制

热重载由 ConfigReloadManager 驱动，包含三个核心环节：

- **文件监听**：通过文件系统事件监听配置目录变更，去抖合并短时间内的多次变更，避免频繁重载。
- **增量重载**：仅解析和重载发生变更的配置文件，不重新加载整个配置目录。
- **校验回退**：新配置校验通过后更新内存；校验或解析失败时回退到备份文件，保持原有配置不变，记录错误日志，daemon 正常运行。

### 配置推送

热重载后，通过事件通道将配置变更通知推送给持有配置引用的已存在会话。会话接收新配置引用后，旧配置对象自动释放。会话本身不重建，已有对话上下文不受影响。

新增会话直接使用最新配置，不受热重载流程影响。

### 去抖策略

文件变更事件可能在短时间内密集到达（如编辑器的多次保存）。ConfigReloadManager 维护去抖窗口，窗口内收到的变更事件合并处理，窗口结束后执行一次重载。

## 数据流

```
文件变更事件
  │
  ├─→ ConfigReloadManager 收到事件
  │     │
  │     ├─ 去抖合并（窗口内后续事件合并）
  │     │
  │     ├─ 读取新文件内容
  │     │
  │     ├─ 校验新配置
  │     │     ├─ 校验失败 → 回退到备份文件，保留旧配置，记录错误（daemon 正常运行）
  │     │     │
  │     │     └─ 校验通过 → 更新内存中的配置缓存
  │     │
  │     └─ 配置更新完成 → 通过事件通道通知已存在会话
  │
  └─→ 已存在会话接收通知 → 更新配置引用 → 新消息使用新配置
```

## 模块关系

- **上游**：ConfigManager（所有配置加载成功后注册热重载监听器，注册后才进入运行状态）、文件系统（触发变更事件）
- **下游**：会话模块（接收配置变更通知，更新配置引用）
- **与 ConfigManager 的关系**：ConfigReloadManager 负责运行时变更处理，ConfigManager 负责启动时加载和写入接口。两者共用 BackupManager 进行回退保护。
- **无关**：processor_chain、tools、skills（不直接依赖热重载，通过上层模块间接使用更新后的配置）
