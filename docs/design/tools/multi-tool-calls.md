# 多工具并行调用

## 概述

多工具并行调用允许 agent 在一个消息中同时发起多个工具调用，减少 round-trip，提升响应速度。系统通过并发安全声明、per-file 互斥队列和配置开关，在利用并行优势的同时防止冲突操作。

## 架构

多工具调用的保护机制分三层：

```
agent 在一个消息中发起多个工具调用
  │
  ├─ 第一层：并发安全声明
  │     → 每个工具通过 is_concurrency_safe 标记自身
  │     → 破坏性工具默认非并发安全（同一文件的写操作需串行）
  │     → 只读工具默认并发安全（多个 Read 可并行）
  │
  ├─ 第二层：per-file 互斥队列
  │     → 同一文件的所有修改操作串行排队
  │     → 不同文件的操作完全并行
  │     → Read 不受队列影响（只读操作无需排队）
  │
  └─ 第三层：配置开关
        → agent 级别可关闭并行调用（保守模式）
        → provider 级别可声明是否支持并行调用
```

### 并发安全声明

每个工具在注册时通过 `is_concurrency_safe` 标记声明自己是否可以在有并行操作时被安全调用：

| 工具 | 并发安全 | 原因 |
|------|---------|------|
| Read | ✅ | 只读，无副作用 |
| Ls | ✅ | 只读，无副作用 |
| Grep | ✅ | 只读，无副作用 |
| Write | ⚠️ | 破坏性，需 per-file 互斥 |
| Edit | ⚠️ | 破坏性，需 per-file 互斥 |
| Bash | ⚠️ | 破坏性，需命令级权限检查 |
| ToolSearch | ✅ | 只读，查询注册表无副作用 |
| sessions_spawn | ✅ | 创建子 session，并发数由 agent 配置 maxChildren 控制 |
| sessions_steer | ✅ | 操作子 session，归属检查在工具层完成 |

⚠️ 标记的工具通过上层保护机制（互斥队列、配额检查、权限引擎）保证安全，不影响其被并行调度。

### per-file 互斥队列

文件修改操作的并发保护核心。同一文件路径的所有 Write 和 Edit 调用串行排队——先到的先执行，后面的等待。不同文件的操作不受影响，完全并行。

互斥队列以真实文件路径为键（解析 symlink），确保同一文件不因路径别名绕过互斥。队列在操作完成后自动清理，无残留。

### 并行调度

agent 在一个消息中发起多个工具调用时，系统执行策略：

- **读取类操作**（Read、Ls、Grep）：无依赖，全部并行执行
- **写入类操作**（Write、Edit）：同一文件的串行排队，不同文件的并行执行
- **混合操作**（Read 文件 A + Edit 文件 B）：不同文件，互不影响，并行执行
- **混合操作**（Read 文件 A + Edit 文件 A）：Read 先于 Edit 执行——Edit 依赖 Read 的 mtime 做 staleness 检查

### 工具运行时标记

工具通过 ToolFlags 声明自身属性，供调度器和权限引擎参考。以下标记与工具并发调度相关：

- **只读标记**：工具是否只读取数据不产生副作用。用于工具索引渲染（显示"只读"标签）。只读工具同时也是并发安全的，但并发安全标记是独立的判断维度
- **并发安全标记**：工具是否可在有并行操作时安全调用。只读工具（Read、Ls、Grep）声明安全，破坏性工具（Write、Edit、Bash）声明不安全
- **资源消耗标记**：工具是否可能消耗较多系统资源（CPU、IO、内存）。声明为资源消耗型的工具包括 Grep（递归搜索）、Git 状态和日志（大型仓库）、Bash（命令执行不可预测）。当前作为信息标记记录，不触发额外行为限制——后续调度器实现后可用于并发配额控制
- **延迟加载标记**：工具的详细描述是否默认不注入 system prompt 一级索引。延迟加载的工具在索引中仅展示名称和危险度标记，LLM 需通过工具发现机制获取完整详情。适用于低频或领域专用工具（Git 操作、Skill 创建等），减少 system prompt 长度

### 配置开关

多工具并行调用有两个配置控制点：

- **Agent 级别**：`parallel_tool_calls` 配置项，默认开启。设为 `false` 时该 agent 的所有工具调用串行执行——适用于对并发安全有严格要求的场景
- **Provider 级别**：声明底层 LLM API 是否支持 `parallel_tool_calls`。不支持的 provider 自动回退到串行模式

### 提示词引导

系统提示词中明确告知 agent 可以并行调用工具：

- 多个读取操作应合并在一个消息中
- 修改同一文件的多个 Edit 应合并到一次 `edits[]` 调用
- 修改不同文件的操作可以并行发起
- 读取文件 A 和编辑文件 B 可以并行
- 标记为 expensive 的工具（Grep、Bash 等）建议控制并行数量，避免过度消耗系统资源

## 数据流

### 并行读取

```
agent 消息：Read(fileA) + Read(fileB) + Grep(pattern, dirC)
  → 三个操作无依赖，全部并行执行
  → 返回三个结果给 agent
```

### 混合读写（同文件）

```
agent 消息：Read(fileA) + Edit(fileA, edits)
  → Read 先执行（Edit 依赖 Read 的 mtime）
  → Edit 获得 mtime → staleness 检查通过 → 执行修改
  → Read 的结果 + Edit 的结果一起返回给 agent
```

### 修改不同文件

```
agent 消息：Edit(fileA, edits) + Edit(fileB, edits)
  → 两个操作目标文件不同 → 并行执行
  → 各自走独立的 per-file 互斥队列
  → 互不干扰，并行完成
```

### 修改同一文件

```
agent 消息：Edit(fileA, edits1) + Edit(fileA, edits2)（不推荐的用法）
  → edits1 先入 per-file 互斥队列 → 执行 → mtime 变更
  → edits2 从队列中取出 → staleness 检查失败（mtime 已变）
  → 返回错误给 agent："文件已被修改，请重新读取后合并为一次 edits[] 调用"
```

## 模块关系

- **上游**：agent 运行时（解析 LLM 返回的多个 tool call，负责并行调度）、LLM provider（声明 `parallel_tool_calls` 能力）
- **下游**：各工具实现（并发安全标记由各工具自行声明）、per-file 互斥队列（Write/Edit 工具调用时入队）
- **无关**：processor_chain（多工具调度在执行层完成，不经过消息处理链路）、IM 适配器（不参与工具调度）
