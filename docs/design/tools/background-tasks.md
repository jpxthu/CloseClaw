# 后台任务

## 概述

后台任务系统为 agent 提供异步命令执行能力。agent 触发长运行命令后无需阻塞等待，命令在后台继续执行，完成后通过通知机制将结果注入 agent 对话流。系统还提供自动后台化、卡死检测和手动后台化等运维能力。

## 架构

后台任务系统由任务管理器、输出管理器和通知系统三部分协同工作：

```
agent 提交后台命令
    │
    ▼
任务管理器 ──→ 创建任务（状态：running）
    │              │
    ├─ 输出管理器 ──→ 输出写入磁盘文件
    │              │
    ├─ 卡死检测器 ──→ 监控输出停滞，识别交互式提示
    │              │
    └─ 进程完成 ──→ 更新状态（completed / failed / killed）
                     │
                     ▼
              通知系统 ──→ 向 agent 注入完成消息
```

### 任务状态机

后台任务的生命周期包含以下状态：

```
        创建
         │
         ▼
      running ────────────────→ killed（被终止）
         │
         ▼
    ┌─────────────┐
    │ completed   │（退出码 0）
    │ failed      │（退出码非 0）
    └─────────────┘
```

状态变迁由进程退出事件驱动，不可逆。killed 状态由外部触发（agent 主动终止或系统清理），一旦进入 killed 状态，后续进程退出通知被忽略。

### 三种后台化路径

#### 显式后台化（run_in_background）

agent 调用 BashTool 时设置 `run_in_background: true`，命令立即转入后台。agent 在返回结果中获得 `backgroundTaskId` 和输出文件路径，无需等待即可继续工作。命令完成后，通知系统向 agent 注入格式化的完成消息。

#### 自动后台化

agent 以前台模式调用 BashTool，命令运行超过阻塞预算（15 秒）时自动转为后台任务。转后台后返回 `assistantAutoBackgrounded: true`，告知 agent 命令仍在执行。此机制确保长时间编译、测试等操作不会让 agent 陷入无响应的等待状态。

自动后台化有命令白名单排除机制：`sleep` 等已知无实际工作的命令不会被自动后台化。

#### 手动后台化

用户通过界面操作（如快捷键）将当前正在阻塞的前台命令全部转为后台，不中断命令执行。适用于用户看到前台命令卡住但不想丢失其产出的场景。

### 输出管理

后台命令的输出写入磁盘文件，而非内存累积。agent 可通过文件读取工具按需查看输出内容。

输出文件的生命周期：
- 任务执行中：持续写入
- 任务完成后：可被读取，随后由系统清理
- 任务被终止：保留当前已写入内容

### 卡死检测

后台任务运行期间，系统监控输出文件的增长速率。当输出在 45 秒内无增长，且尾部内容匹配交互式提示特征（如 `(y/n)`、`Continue?`、`Are you sure?` 等），系统判定命令被交互式提示卡住，向 agent 注入警告通知，建议终止任务并以管道输入方式重新运行。

仅慢速但有效的命令（如大型构建、长时间计算）不会触发卡死检测——检测只在尾部内容疑似交互式提示时才告警。

### 通知机制

任务完成或卡死时，系统生成结构化通知消息注入 agent 的消息队列。通知包含：

- 任务标识（taskId）
- 输出文件路径
- 完成状态（completed / failed / killed）
- 可读摘要（"Background command '&lt;command&gt;' completed"）

通知有两种优先级：
- **next**：在下一轮对话中尽早注入，适用于卡死警告等需要 agent 立即响应的场景
- **later**：在合适时机注入，适用于普通完成通知

通知带去重保护：同一任务只发一次通知，防止消息队列中重复注入。

### 提示词引导

为防止 agent 在后台任务执行期间做无意义的轮询，系统提示词中明确注入以下行为约束：

- **不需要立即检查输出**：告知 agent 后台命令完成后会自动收到通知，无需主动查询状态
- **不主动 poll**：agent 不应调用进程查询工具去检查后台任务是否完成。push-based 通知机制保证 agent 在下一轮对话中自动看到结果
- **使用 run_in_background 的建议场景**：预计运行时间超过 10 秒的命令应显式后台化，不阻塞 agent 的后续工作

工具描述中也直接嵌入此指引，确保 agent 在每次决定是否后台化时能看到行为约束。

## 数据流

### 任务创建与完成

```
BashTool.call() → run_in_background: true
  → 任务管理器创建后台任务
    → 状态设为 running
    → 启动卡死检测定时器
    → 命令异步执行，输出写入磁盘
      → 进程退出
        → 读取退出码
        → 更新状态：退出码 0 → completed，非 0 → failed
        → 取消卡死检测
        → 通知系统生成完成消息
          → 注入 agent 消息队列（优先级 later）
        → 输出文件标记可清理
```

### 自动后台化流程

```
BashTool.call() → 前台模式
  → 命令开始执行，阻塞预算计时器启动
    → 15 秒内完成：正常返回结果
    → 15 秒未完成：
      → 命令转后台（进程继续运行）
      → 任务状态设为 running（isBackgrounded: true）
      → 返回 assistantAutoBackgrounded: true + backgroundTaskId
      → agent 被告知命令仍在后台运行
      → 后续完成时走任务完成通知流程
```

### 卡死检测与通知

```
后台任务运行中
  → 定时检查输出文件大小（每 5 秒）
    → 输出在增长：重置计时器
    → 输出停滞达到 45 秒：
      → 读取输出尾部内容
        → 不匹配交互式提示特征：继续监控（可能是慢任务）
        → 匹配交互式提示特征：
          → 向 agent 注入警告通知（优先级 next）
          → 建议终止任务并重试（如用管道输入绕过交互）
          → 此任务不再重复检测
```

### 手动后台化

```
用户通过界面操作触发
  → 遍历当前所有前台任务
    → 逐个转后台，进程继续执行
    → 任务状态设为 running（isBackgrounded: true）
    → 启动卡死检测定时器
    → 返回 backgroundedByUser: true + backgroundTaskId
    → 后续完成时走任务完成通知流程
```

### 停止时的清理

当 session 被停止（用户 `/stop`、父 session 级联停止、或系统优雅关闭）时，该 session 的所有后台任务进程被终止。任务状态标记为 killed，已产生的输出保留在磁盘文件中。终止后的进程退出通知被忽略——killed 状态不可逆。

清理通过两种终止适配器实现，前台和后台走不同路径：

- **前台终止适配器**：持有子进程的共享句柄，停止时直接向进程发送终止信号。子进程被立即终止，已累积的输出保留
- **后台终止适配器**：通过异步运行时桥接调用后台任务管理器的终止方法。后台任务管理器负责定位并终止对应进程，已写入磁盘的输出文件保留

清理由 Session 模块的级联停止机制驱动——工具在执行期间将对应的终止适配器注册到会话，完成后注销。级联停止时 Session 遍历所有已注册的终止适配器依次执行清理。

## 模块关系

- **上游**：BashTool（触发后台执行、自动后台化）、Session 模块（停止时触发进程清理）
- **下游**：文件系统（输出持久化）、消息队列（通知注入）、agent 会话（通知被 agent 在下一轮对话中看到）
- **无关**：权限引擎（权限检查在 BashTool 层完成，后台系统不重复检查）、processor_chain（通知注入走消息队列，不走出站处理链）
