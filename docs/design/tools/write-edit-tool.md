# Write/Edit 工具

## 概述

Write 和 Edit 是 agent 修改文件系统的两个工具。Write 负责创建和覆盖写入，Edit 负责精确文本替换。两者共享同一套并发安全保护机制——per-file 互斥队列确保同一文件的操作串行执行，不同文件的操作完全并行。

## 架构

两个工具各有独立执行路径，但在文件突变前共用互斥队列：

```
agent 调用 WriteTool / EditTool
  → 参数校验
    → per-file 互斥队列入队（同一文件串行等待）
      → 写入前 staleness 检查（mtime 比对）
        → 执行写入 / 文本替换
          → 写入后校验（可选 readback）
            → 返回结果给 agent
```

### Write 工具

#### 参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `path` | string | 是 | 文件路径，绝对路径或相对于工作目录 |
| `content` | string | 是 | 要写入的文本内容 |

Write 工具创建或覆盖写入指定文件。文件不存在时自动创建（含父目录）。文件已存在时直接覆盖，不保留原内容。覆盖是破坏性操作——工具标记为破坏性工具。

### Edit 工具

#### 参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `path` | string | 是 | 目标文件路径 |
| `edits` | array | 是 | 一或多个替换操作，每个包含 `oldText` 和 `newText` |
| `edits[].oldText` | string | 是 | 要替换的原始文本，必须在文件中精确匹配且唯一 |
| `edits[].newText` | string | 是 | 替换后的文本 |

Edit 工具在一次调用中完成对同一文件的一到多个精确文本替换。所有 edits 都对着调用时的原始文件内容做匹配（non-incremental matching），匹配完成后再统一应用修改。

工具标记为破坏性工具。

#### 匹配与应用策略

**Non-incremental matching**：所有 `edits[].oldText` 在应用任何修改之前，先对原始文件内容完成全部匹配。这样即使多个 edits 修改了文件的相邻区域，匹配也不会受前一个修改的影响。

**逆序应用**：匹配完成后，按 `oldText` 在文件中的位置**从后往前**依次应用替换。因为是从后往前，后面的修改不会改变前面修改在文件中的偏移位置。

**唯一性检查**：每个 `oldText` 在文件中必须恰好匹配一次。匹配零次则报错（文件内容与 agent 认知不一致）；匹配多次也报错（无法确定要修改哪一处）。`replace_all` 参数（Boolean，可选）可以显式允许全量替换来覆盖多次匹配的场景。

**重叠检查**：任意两个 edits 的匹配区域有重叠则拒绝，防止一个替换破坏另一个替换的内容定位。

#### 模糊匹配

精确匹配失败时，进入模糊匹配降级：Unicode 规范化（NFC/NFD）、弯引号与直引号互转、行尾空白去除。模糊匹配成功则按模糊匹配的位置应用修改；不成功再报错。

模糊匹配只影响匹配阶段——`newText` 中的引号风格保留原样写入，不做自动转换。

#### 写入前 staleness 检查

Edit 工具执行修改前，检查文件的当前 mtime 是否与 agent 上次 Read 该文件时的 mtime 一致：

- **一致**：文件未被外部修改，agent 的认知与磁盘状态同步，继续执行
- **不一致**：文件在 agent 读取后被外部修改（如其他 agent、后台任务、用户手动编辑），拒绝本次修改，返回错误提示 agent 重新读取文件

Write 工具不依赖 Read 历史，但同样在写入前检查 mtime——如果文件在决定写入到实际写入之间被修改，Write 也拒绝执行，要求 agent 确认意图后重试。

#### 写入保护

同一文件的所有 Write 和 Edit 操作通过 per-file 互斥队列串行化——这是个基于真实文件路径的排队机制。不同文件的操作完全并行，同一文件的操作按到达顺序依次执行。

互斥队列解析 symlink 到真实路径，防止同一文件通过不同路径名绕过互斥。

#### 写入后恢复

工具调用因外部原因（超时、连接中断）异常终止，但写入已完成时，通过 readback 检测文件是否已被修改：如果文件内容匹配预期，视为写入成功，返回成功结果而非报错。

## 数据流

### Write 流程

```
agent 调用 WriteTool(path, content)
  → 参数校验（path 不为空、content 有效性）
  → per-file 互斥队列入队 → 等待该文件的队列清空
  → staleness 检查（文件存在时，mtime 与上次 Read 比对）
    → 不一致 → 拒绝，返回"文件已被外部修改"错误
  → 写入文件（创建 + 写入，自动创建父目录）
  → 返回成功结果给 agent
```

### Edit 流程

```
agent 调用 EditTool(path, edits[])
  → 参数校验（每个 edit 的 oldText 不为空、oldText ≠ newText）
  → per-file 互斥队列入队 → 等待该文件的队列清空
  → staleness 检查（mtime 与上次 Read 比对）
    → 不一致 → 拒绝
  → 读取当前文件内容
  → 对每个 edit.oldText 做匹配（先精确、再模糊）：
    → 零次匹配 → 报错："未找到指定文本，文件可能已被修改"
    → 多次匹配 → 报错："找到 N 处匹配，请缩小 oldText 范围或使用 replace_all"
  → 重叠检查：任意两个匹配区域重叠 → 拒绝
  → 按匹配位置从后往前依次应用替换
  → 写入修改后的内容到文件
  → 返回成功结果（含实际替换的 edit 数量）
```

### 写入冲突的典型场景

```
场景 1：agent 连续两次 Edit 同一文件
  → 第一次 Edit 入队执行，第二次 Edit 入队等待
  → 第一次完成后，第二次开始执行
  → staleness 检查：mtime 已变（第一次修改导致）→ 拒绝
  → agent 收到错误，重新 Read → 再次 Edit

场景 2：agent 同时 Read 文件 A 和 Edit 文件 B
  → 不同文件，互斥队列不冲突 → 并行执行

场景 3：agent 在同一消息中同时 Edit 文件 A 两次
  → per-file 互斥队列串行化两次调用 → 第一次成功
  → 第二次 staleness 检查失败 → 拒绝
  → 提示 agent 合并为一次 edits[] 调用
```

## 模块关系

- **上游**：agent 运行时（调度工具调用、传递参数）
- **下游**：文件系统（直接读写文件）、Read 工具（提供 staleness 检查所需的 mtime）、会话上下文（写入结果注入 agent 对话流）
- **无关**：权限引擎（写入权限在工具调用前由权限引擎校验，Write/Edit 自身不做权限判断）、processor_chain（不参与消息出站处理）

### 与模块内其他子功能的关系

- **Read 工具**：Edit 的 staleness 检查依赖 Read 工具记录的文件 mtime。agent 必须先 Read 文件才能 Edit
- **多工具调用**：per-file 互斥队列是多工具并行调用的核心保护机制——详见 multi-tool-calls.md
