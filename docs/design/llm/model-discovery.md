# 模型发现

## 概述

模型发现子系统从供应商 API 动态获取可用模型列表，结合内置知识库和本地缓存，自动填充每个模型的推荐参数（上下文窗口、最大输出、是否为推理模型等），减少用户手工配置。

## 架构

模型发现由三层机制配合完成：

```
用户输入凭据
  → 模型发现流程
    ├─ 本地缓存：TTL 1 小时，命中即返回
    ├─ 动态探测：调用供应商 /models API（最新数据，10s 超时）
    └─ 内置知识库：静态模型数据，API 不可用时的回退
  → 模型列表 + 推荐参数
```

### 动态探测

每个供应商提供 `fetch_model_list` 接口，向各自 `/models` 端点发送带凭据的 HTTP GET 请求。探测超时 10 秒，网络异常或 API 返回异常时进入回退。

### 本地缓存

模型列表缓存到本地文件。缓存键由供应商 ID 和凭据指纹组成（指纹为凭据明文的前 4 个字符，凭据变更后缓存自动失效），TTL 为 1 小时。缓存命中且未过期时直接返回，不调 API；过期时重新调用 API 并更新缓存。

### 内置知识库

知识库为每个供应商维护已知模型的基础参数：上下文窗口大小、最大输出 token 数、默认温度、是否推理模型、输入类型。这些数据作为动态探测失败时的回退，也用于填充 API 未返回的字段。

对于已知模型标识符的单模型参数查询（如读取配置中的模型名后获取其能力参数），可直接查知识库获取，无需走完整的发现流程。

### 模型信息结构

每个发现的模型包含：模型 ID、显示名、上下文窗口 token 数、最大输出 token 数、默认温度值、是否为推理模型、支持的输入类型。

## 数据流

```
查询可用模型（供应商 ID + 凭据）
  → 计算缓存键 → 查本地缓存
    → 命中且未过期：返回缓存
    → 未命中或已过期：
      → 调用供应商 /models API（10s 超时）
        → 成功：解析模型列表 → 写入缓存 → 返回
        → 失败/超时：查询内置知识库 → 返回（标注回退来源）
```

## 模块关系

- **上游**：provider-config-wizard（向导中触发模型发现）
- **下游**：各供应商 API（HTTP 调用 /models 端点）、本地文件系统（缓存读写）
- **无关**：Session 层、Rendering Layer（模型发现是配置阶段的子系统，与运行时对话路径无关）
