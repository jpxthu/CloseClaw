# 模型发现

## 概述

模型发现子系统从供应商 API 动态获取可用模型列表，结合内置知识库，自动填充每个模型的推荐参数（上下文窗口、最大输出、是否为推理模型等），减少用户手工配置。

## 架构

模型发现由两层机制串行配合完成：先调供应商 /models API 动态探测，API 不可用时回退到内置知识库。

1. 动态探测 — 调供应商 /models API 获取最新模型列表（10s 超时，瞬态错误最多重试 3 次）
   - 成功 → 过滤未知模型 → 用知识库填充能力参数 → 返回
   - 失败 → 进入步骤 2
2. 内置知识库 — 返回该供应商所有已知模型，标注回退来源

对于已知模型标识符的单个参数查询，可直接查知识库获取，无需走完整发现流程。

### 动态探测

每个供应商提供模型列表查询接口，向各自 `/models` 端点发送带凭据的 HTTP GET 请求。探测超时 10 秒，网络超时等瞬态错误最多重试 3 次（指数退避约 1s → 2s → 4s，含抖动）。认证失败、计费错误等不可恢复错误不重试，直接进入回退。

### 内置知识库

知识库为每个供应商维护已知模型的基础参数：上下文窗口大小、最大输出 token 数、默认温度、是否推理模型、推理档位模式、推荐协议、支持的输入类型。这些数据作为动态探测失败时的回退，也用于填充 API 未提供的能力参数——知识库是模型能力参数的唯一权威来源。`/models` API 只返回模型 ID 和所有者信息，不包含任何能力参数。系统仅支持知识库已知模型，API 返回的未知模型（知识库中不存在的模型 ID）被过滤，不返回给调用方。

知识库按供应商拆分，每个供应商一个数据文件，编译期内嵌到二进制中。文件内按模型 ID 索引。新增或更新模型参数时只修改对应数据文件，不需要改动代码。

对于已知模型标识符的单模型参数查询（如读取配置中的模型名后获取其能力参数），可直接查知识库获取，无需走完整的发现流程。

### 模型信息结构

每个发现的模型包含：

- 模型 ID
- 显示名（API 返回 display_name 则使用，否则从模型 ID 格式化派生）
- 上下文窗口 token 数
- 最大输出 token 数
- 默认温度值
- 是否为推理模型
- 推理档位模式
- 推荐协议
- 支持的输入类型（text / image）

## 数据流

```
查询可用模型（供应商 ID + 凭据）
1. 调用供应商 /models API（10s 超时，瞬态错误最多重试 3 次）
   - 成功 → 解析模型列表 → 过滤未知模型 → 用知识库填充能力参数 → 返回
   - 不可恢复错误（认证失败、计费错误等）→ 步骤 2
   - 重试耗尽仍失败 → 步骤 2
2. 查询内置知识库 → 返回（标注回退来源）

单模型参数查询（已知模型 ID）
  → 直接查知识库 → 返回
```

## 模块关系

- **上游**：provider-config-wizard（向导中触发模型发现）
- **下游**：各供应商 API（HTTP 调用 /models 端点）
- **无关**：Session 层、Rendering Layer（模型发现是配置阶段的子系统，与运行时对话路径无关）
