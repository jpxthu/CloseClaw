# 审批工作流

## 概述

当 Agent 操作被权限引擎判定为 Deny 时，不直接拒绝——进入审批工作流，由 Owner 决定放行、拒绝或将操作加入白名单。

## 架构

### 审批分流

不同场景的 Deny 处理方式不同：

```
权限引擎 Deny
  │
  ├─ 子 Agent 操作 → 静默 Deny（不向上推送）
  ├─ Heartbeat 任务 → 按任务配置：静默跳过 / 通知 Owner / 走审批
  └─ 其他场景 → 推送 Owner 审批
```

### 审批队列

审批请求由守护进程内的审批队列管理，是内存数据结构：

- **入队**：工具调用被 Deny（非子 Agent、非 Heartbeat-skip）→ 创建审批请求入队 → 工具调用返回等待状态，不阻塞 Agent 后续执行
- **去重**：入队前检查同一 caller + 同一操作是否已有等待中的请求 → 有则直接拒绝，不重复入队
- **回调**：Owner 审批后，通过请求 ID 关联回调，通知 Session 继续执行或返回 Deny
- **进程重启**：重启时清空队列，所有等待请求以 Deny 结束

### 审批信息

推送 Owner 的审批信息包含：发起操作的 Agent ID、当前 User ID、操作类型和参数（路径、命令等）、风险等级提示、审批请求唯一 ID。

风险等级不影响是否触发审批（符合 Deny 条件即触发），但影响 Owner 可选的审批操作：高风险和严重风险操作不可选"加入白名单"，只能单次放行或拒绝。

### Owner 审批权力

所有审批请求只推送给 Owner。非 Owner 用户没有审批权力。Owner 可选择：

- **单次放行**：仅本次操作放行，不持久化规则，后续相同操作仍需重新审批
- **加入白名单**：写入权限规则，后续同类操作自动放行。高风险和严重风险操作不可选此项
- **拒绝**：操作被 Deny

Owner 可以按 Agent 维度和 User 维度分别决定：只给这个 Agent 放行、只给这个 User 放行、或两者都放行。Owner 自己调用时只审 Agent 维度。

### 高危操作

标注为高风险或严重风险的操作有额外限制：永远只能走单次审批，不能被加入白名单。高危操作以硬编码名单维护，初期包括版本控制目录读写、权限配置文件读写、系统配置文件写入、破坏性删除命令等。

## 数据流

```
Agent 工具调用被 Deny
  │
  ├─ 子 Agent / Heartbeat-skip → 静默 Deny（结束）
  │
  └─ 需审批场景
        │
        ▼
     创建审批请求（记录 RuleSet 版本号快照）
        │
        ▼
     入队（去重检查：同 caller + 同操作 → reject）
        │
        ▼
     推送 Owner 审批卡片（通过外部层 → IM Adapter 渲染）
        │
        ▼
     Owner 响应（被 Gateway 硬拦截，不进入 Agent session）
        │
   ┌────┼────┐
   ▼    ▼    ▼
 单次OK 白名单 拒绝
   │    │    │
   │    ▼    │
   │  写入规则  │
   │    │    │
   ▼    ▼    ▼
 通过回调恢复 Session 执行 / 返回 Deny
```

关键分支：
- Owner 在审批期间修改了权限规则 → 审批响应时使用入队时记录的版本快照评估，不受中途变更影响
- 进程重启 → 清空队列，所有等待请求以 Deny 结束

## 模块关系

- **上游**：权限引擎（产出 Deny 需审批的结果）
- **下游**：Gateway/外部层（推送审批卡片给 Owner IM）、Agent Session（审批完成后回调恢复执行）
- **相关**：权限配置管理（白名单选项触发规则写入）
- **无关**：IM Processor（审批卡片的平台渲染由外部层 Adapter 处理）
