# IM Adapter 需求

## 概述

IM Adapter 模块满足 CloseClaw 跨消息平台运行的核心诉求——用户和 Agent 可以通过飞书、Discord、Telegram 等多种 IM 平台与 CloseClaw 交互，消息格式在各平台间无缝转换，无需感知底层平台差异。

## 功能需求

### F1. 多平台插件化适配

CloseClaw 需要支持多种 IM 平台，每个平台以独立插件形式接入。新增平台时无需改动 Gateway 等核心模块代码。平台支持可按需启用——默认只启用配置文件中有明确配置的平台，未配置的平台不加载。

每个平台插件自包含，内部包括：
- 协议适配：将平台原生消息格式转为统一的内部消息结构，将内部渲染结果通过平台 API 发送
- 格式渲染：将 Agent 的结构化输出转换为平台原生消息格式（富文本卡片、纯文本等）
- 平台工具：该平台特有的工具能力（如飞书的日历、文档等）

插件在系统启动时加载。未在配置中启用的平台不占用运行时资源。新平台 = 新增自包含目录，Gateway 不需要修改代码。

### F2. 入站消息归一化

来自不同 IM 平台的消息在进入系统内部前，需要被归一化为统一的中间消息结构，使下游模块（Processor Chain、Gateway、LLM）不感知平台差异。归一化工作由各平台插件在入站解析时完成。

归一化后的消息包含平台标识、发送者、会话 ID、消息正文等标准化字段。各平台特有的元数据（如 API 字段名、租户标识等）不进入归一化结构。

引用/回复消息：若平台支持消息引用功能，被引用消息的文本内容需截断后拼接到消息正文前（截断至约 500 字符），以 markdown 引用格式呈现。不传递独立的引用消息字段。

非文本消息（图片、文件、语音等）：记录日志后丢弃，当前不进行语义理解处理。

### F3. 出站消息格式自动选择

Agent 输出消息在发送到 IM 平台前，需根据内容特征自动选择输出格式：

- 纯文本、无格式标记、无换行附 → 轻量文本消息
- 含 markdown 格式（标题、粗体、斜体、代码块、列表、引用、链接、分割线）或换行符 → 富格式消息
- 含 Thinking/Tool 相关块 → 富格式消息
- 含交互指令 → 富格式消息（含按钮等交互元素）

格式选择由渲染器自动完成，用户和 Agent 无需手动指定输出格式。

### F4. 流式增量渲染

Agent 流式输出时，消息应逐步渲染并发送到 IM 平台，使用户在 Agent 思考过程中即可看到已生成的内容，无需等待完整响应。

流式渲染时，文本内容以完整的语义行为单位输出——遇到句末标点或换行符时立即发送当前行。代码块内部按换行符逐行输出。若缓冲内容长时间未遇到句末标点，应在约 100 字符时强制输出，避免长时间无响应。

Thinking 块和 Tool 块不参与流式行缓冲——累积完整内容后一次性交付平台格式渲染器处理。图片、音频、文件等多媒体块不参与流式渲染，直接交由平台格式渲染器。

流错误不产生增量输出，由 Gateway 统一处理。

### F5. 代码块语法高亮

消息中的代码块（三反引号包裹的代码段）需要按编程语言进行语法高亮渲染。各平台按自身能力选择渲染策略：

- 平台原生支持代码块高亮的，使用平台原生能力（如飞书卡片 markdown 代码块）
- 终端等环境，通过 ANSI 颜色码实现语法高亮
- 平台不支持特殊渲染的，保留代码块标记以纯文本形式展示

语言标注从代码块标记中提取（如 ` ```rust ` 表示 Rust 高亮），各平台自行决定支持的语言列表。

### F6. 飞书平台完整支持

飞书平台需提供完整的消息交互能力。

入站方面：解析飞书 webhook 事件，支持 text 和 post 两种消息类型。post 类型消息需展开富文本结构（有序/无序列表、文本样式、图片占位符等）为 markdown 文本。话题消息提取话题 ID 用于回复定位，不参与会话路由。

出站方面：将 Agent 输出渲染为飞书消息格式。首行 `#` 标题提取为卡片标题（蓝色模板）。正文中的 markdown 元素（标题、粗体、斜体、行内代码、代码块、引用、列表、链接、分割线）渲染为飞书原生对应的卡片元素。Thinking 块以折叠或其他可区分形式展示。Tool 调用的执行状态和结果以可读的格式展示。

交互方面：按钮指令以飞书按钮组件形式渲染——首个按钮为主要样式，其余为默认样式，按钮平铺排列。

平台工具方面：飞书插件提供以下工具能力，均支持延迟加载（首次调用时才初始化）：
- IM 消息操作（发送、撤回、编辑、表情回应等）
- 日历管理
- 任务管理
- 多维表格操作
- 文档操作
- 云盘操作
- 电子表格操作

### F7. 渲染与发送分离

Agent 输出在发送前需要经过渲染和发送两个独立步骤。渲染产生平台特定的消息格式数据，发送是产生副作用的网络操作。

渲染和发送之间可以插入中间件（如审计日志、频率限制等），Gateway 在此处拥有控制权。

## 非功能需求

### 性能

- 流式渲染：Agent 首个文本块到达后，首行输出应在 200ms 内送达用户
- 行缓冲强制输出阈值：约 100 字符，避免流式输出长时间无响应

### 可扩展性

- 新增 IM 平台不修改核心模块（Gateway 等）代码
- 平台插件默认可被编译期自动发现
- 不在标准插件目录下的特殊插件（如终端）支持显式注册

### 稳定性

- 单个平台插件的初始化失败不影响其他已启用平台的正常运行
- 平台 API 调用失败时，系统正常降级，不导致进程崩溃

### 安全性

- 平台凭证（token 等）由各平台插件自行管理，不在模块间传递
- 被引用消息的文本截断限制（500 字符），防止超长引用内容占用上下文
