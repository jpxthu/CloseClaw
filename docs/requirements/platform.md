# Platform 需求

## 概述

Owner 将 CloseClaw 部署到不同操作系统时，不需要关心 OS 差异——进程管理、配置目录、终端行为、文件路径都由 platform 模块统一处理。其他功能模块不感知操作系统差异，始终使用平台无关的接口。

## 功能需求

### F1. 跨平台进程生命周期管理

系统在 Linux、macOS、Windows 上提供一致的启停行为：启动时创建 PID 文件以追踪进程实例；停止时通过操作系统标准的终止机制关闭进程（Linux/macOS 使用信号，Windows 使用进程终止 API），而非强制终止进程。

> **交叉引用**：Owner 通过 daemon 命令启停服务。详见 [cli §F3](cli.md)（Daemon 生命周期管理）。

### F2. 操作系统标准配置目录

Owner 在任意操作系统上首次启动 CloseClaw 时，配置文件自动创建在操作系统标准配置目录下，无需手工指定路径。Linux/macOS 使用 `~/.closeclaw`，Windows 使用 `%APPDATA%\closeclaw`。

### F3. 终端能力自适应

使用 CLI 交互的 User 获得与终端能力匹配的显示体验。当终端支持 ANSI 转义序列时，系统渲染彩色与富文本输出；当终端不支持时，自动降级为纯文本输出，不出现乱码或转义序列原样显示。

### F4. 跨平台文件路径处理

Owner 和 User 使用自己操作系统习惯的路径格式——Linux/macOS 用 `~` 简写 home 目录、`/` 为路径分隔符，Windows 用 `%APPDATA%` 环境变量、`\` 为路径分隔符。系统自动展开路径简写并按平台转换分隔符，Owner 和 User 不需要了解内部如何统一路径表示。

### F5. 系统差异隔离

其他模块不依赖任何操作系统特定的接口或行为。所有操作系统差异（进程信号、文件路径分隔符、环境变量格式、配置目录惯例、终端能力检测）统一由 platform 模块消化，向上层模块暴露平台无关的抽象接口。上层模块不需要针对不同操作系统编写分支逻辑。

## 非功能需求

- **启动速度**：进程创建和 PID 文件写入不应成为 daemon 启动的瓶颈，开销可忽略，不影响 daemon 启动体验
- **跨平台一致性**：同一功能在 Linux、macOS、Windows 上的行为语义一致，Owner 切换操作系统时不需要重新学习或调整配置
- **稳定性**：进程终止信号/API 调用必须可靠，不产生僵尸进程或 PID 文件残留
