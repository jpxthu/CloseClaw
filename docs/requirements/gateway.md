# Gateway 需求

## 概述

Gateway 是消息路由中枢。User 通过不同 IM 平台发送的消息由 Gateway 统一接入、识别类型、按规则路由到斜杠指令或 LLM 对话，并将所有回复经统一出站管道处理后发回对应平台。

## 功能需求

### F1. 多平台消息统一接入

- User 可以通过飞书、Discord、Telegram、CLI 等多种渠道发送消息，系统对不同平台的消息以统一结构处理
- 所有平台的消息走同一条入站处理链路，平台差异在接入层消除

### F2. 入站消息预处理

- 入站消息内容做平台无关的文本标准化：去除控制字符和 ANSI 转义序列、压缩连续空行、去行尾空格
- 空文本消息在接入阶段丢弃，不进入后续处理
- 每条入站消息记录完整审计日志，包含原始内容和来源信息

### F3. 消息类型识别与非文本处理

- 系统能识别文本、图片、文件、音频等多种消息类型
- 图片、文件、音频等非文本消息给出明确的"暂不支持该消息类型"提示后结束处理，不进入 LLM 对话

### F4. 普通消息路由到对话

- 不以 `/` 开头的文本消息进入 LLM 对话流程
- 系统自动根据消息来源（平台、发送者、会话标识）查找已有会话或创建新会话
- 若对应 Session 正忙（LLM 运行中或工具执行中），新消息进入排队，向 User 回复"⏳ 正在排队..."
- 若 Session 处于归档状态，系统自动触发恢复流程，向 User 回复"正在恢复会话..."
- 会话路由失败时，向 User 回复明确的错误提示

### F5. 斜杠指令拦截与分派

- User 可以用 `/` 开头发送斜杠指令，指令在 Gateway 层被拦截，不进入 LLM 对话、不追加到对话历史
- `/approve`、`/deny` 仅 Owner 可用，用于处理审批请求
- Immediate 类指令（如 `/stop`、`/status`、`/help`）绕过 Session 排队，即使 LLM 运行中也能立即响应
- 非 Immediate 斜杠指令在 Session 忙时进入排队等待

### F6. 入站消息队列与过载保护

- 高并发入站消息由有界缓冲队列暂存，按先进先出顺序处理
- 队列满时拒绝新消息，立即回复 User"服务繁忙，请稍后重试"
- 系统重启时队列清空，队列满期间被拒绝的消息由 IM 平台重试机制补偿

### F7. 出站消息统一处理

- LLM 回复和斜杠指令回复均走同一条出站处理链路
- 回复内容按目标平台的格式渲染后发送，不同平台使用各自的渲染规则
- 出站消息发送后持久化记录到会话存档

## 非功能需求

- 入站消息队列满时，"服务繁忙"拒绝响应必须在 2 秒内送达 User
- 单条消息从接入到路由决策完成的处理延迟不超过 500ms
- 系统重启后未处理的消息由 IM 平台 webhook 重试机制保证不丢失
