# 权限系统 需求

## 概述

Owner 需要一套身份型访问控制系统：定义"谁能以什么身份做什么"，覆盖 Agent 工具调用和用户斜杠指令两大类操作，并在操作被拦截时提供可控的审批通道。系统自身行为（会话管理、消息路由、系统提示构建等）不经过权限检查。

## 功能需求

### F1. 身份体系

Owner 需要系统区分两种独立身份：

- **Agent 身份**：每个 Agent 有独立的权限边界。新建 Agent 默认只能收发消息，无文件读写、无命令行、无网络访问。Agent 的操作始终在某个 User 上下文中执行——不存在无 User 上下文的操作。
- **User 身份**：每个 User 通过 IM 渠道 + IM User ID 的组合绑定到系统。一个 User 可以绑定多个 IM 渠道（如飞书 Open ID + Telegram User ID），映射为同一个人。新建 User 默认无任何权限，收发消息也需 Owner 显式授予。不区分用户组，所有权限以个人为单位。

User 上下文的来源由场景决定：IM 消息来自发送者、CLI 调用默认为 Owner、定时任务使用任务配置中指定的 User ID、子 Agent 继承父 Agent 的 User 上下文。

Owner 的 User ID 固定为 `owner`，额外拥有 CLI 渠道的访问方式。

### F2. 权限维度

Owner 需要八个独立管理的权限维度：

- **读**：控制 Agent 可读取的文件和目录范围。
- **写**：控制 Agent 可写入的文件和目录范围。读写共用同一套文件级权限检查机制。
- **命令行**：控制 Agent 可执行的命令。有权限的命令在沙盒外执行，无权限的命令和所有脚本在沙盒内执行。命令白名单支持放行和阻止两种参数。
- **工具调用**：控制 Agent 能否调用某个工具。工具内部的文件读写直接受读/写维度的判定覆盖，不重复判定。
- **收发消息**：控制 Agent 或 User 能否收发消息。LLM 输出的普通文本消息不经过工具模块，只有工具调用格式的输出才触发权限检查。
- **网络访问**：控制 Agent 可访问的网络资源（主机、端口）。默认拒绝所有网络访问。
- **跨 Agent 通信**：控制 Agent 之间能否通信（spawn、消息传递等）。默认拒绝。
- **配置写入**：控制 Agent 能否写入系统配置文件。默认拒绝，且此维度永远高风险——只能走单次审批，不能被加入白名单。

### F3. 权限决策模型

Owner 需要 Agent 维度和 User 维度的交集决策：操作只有在双方都允许时才放行，任一方拒绝即拒绝。

```
Agent 维度 Allow AND User 维度 Allow → 放行
任一方 Deny → 直接 Deny
```

每个 Agent-User 组合自动获得其专属 workspace 路径（`{数据根目录}/workspaces/{Agent ID}/{User ID}/`）的读写权限。此路径由系统自动创建和管理，Agent 即使权限规则未覆盖此路径也可正常读写。此强制授权不受任何 Deny 规则影响。

### F4. Owner 特权

Owner 在权限决策中享有以下特权：

- **User 维度豁免**：Owner 作为调用方时，跳过所有 User 维度规则，仅评估 Agent 维度。这意味着 Owner 不会被 User 规则拦截，只受该 Agent 自身能力边界的限制。
- **权限配置独占**：权限配置文件的写入仅限 Owner。Agent 无法通过任何手段读取或写入权限配置。
- **审批独占**：所有审批请求只推送给 Owner，非 Owner 用户没有审批权力。Owner 可对审批请求做出三种决策：单次放行（仅本次操作）、加入白名单（持久化规则）、拒绝。

### F5. 审批工作流

当 Agent 操作被权限系统拒绝时，Owner 不希望直接阻断——需要一个可控的审批通道：

- **审批触发**：操作被拒绝时按场景分流处理（子 Agent 场景见 F9）——定时任务的拒绝按任务配置处理（静默跳过/通知/进入审批流程）、其余场景自动创建审批请求并推送给 Owner。
- **去重**：同一个调用方 + 同一个操作的重复申请直接拒绝，不重复推送审批。
- **审批信息**：推送的审批信息需包含发起操作的 Agent ID、当前 User ID、操作类型和参数、风险等级提示、审批请求唯一 ID。
- **Owner 决策**：Owner 可选择单次放行（仅本次操作，不持久化）、加入白名单（写入永久规则）、或拒绝。
  - 审批非 Owner 调用方的操作时：Owner 可按 Agent 维度和 User 维度分别决策——仅放行触发操作的 Agent、仅放行当前 User 上下文、或两者均放行。
  - Owner 自身调用被拦截时：仅审 Agent 维度（User 维度已豁免）。
- **审批快照**：审批发起时记录当前的权限规则版本。Owner 审批响应时使用发起时的规则版本进行评估，中途的权限变更不影响本次审批结果。
- **等待机制**：Agent 一轮调用多个工具时，有权限的工具可先执行；被拒绝的工具调用进入等待状态，等待 Owner 审批。审批有默认等待时间，超时未响应则自动返回拒绝信息。Owner 审批完成后系统自动回调通知结果。系统异常中断时清空所有等待中的审批请求，并向对应 Agent 返回拒绝信息，避免调用端悬挂。

### F6. 权限配置管理

Owner 需要灵活高效地管理权限规则：

- **修改方式**：Owner 通过斜杠指令或审批白名单两种方式修改权限规则。两种方式均不在 Agent Session 中执行，而是由系统直接处理。任何修改操作在执行前验证调用方是否为 Owner。
- **热加载**：权限规则变更后立即生效，无需重启 Agent 或系统。下一次权限检查即使用新规则。
- **新建 User**：非 Owner 用户创建需 Owner 审批。新建 User 默认无任何权限，包括不能与任何 Agent 通信。创建后 Owner 需另行授予权限，用户才能开始交互。

### F7. 高风险操作管控

Owner 需要系统对高风险操作施加额外限制。风险等级分为"高风险"和"严重风险"两级，统称高风险操作：

- 标注为高风险或严重风险的操作永远只能走单次审批，不能被加入白名单。
- 风险等级影响 Owner 审批时的可选操作：高风险和严重风险操作的审批卡片不显示"加入白名单"选项。
- 高风险操作列表由系统预定义，至少包含：版本控制目录读写、权限配置文件读写、系统配置文件写入、破坏性删除命令等。

### F8. 安全边界

Owner 需要以下安全保障：

- **配置保护**：权限配置文件不在 Agent 可访问的路径下，Agent 无法读取或写入。权限配置目录整体对 Agent 禁止访问。
- **不可覆写规则**：权限系统在安全设计上不存在以下操作入口：禁用 Owner 身份校验、绕过权限系统、由非 Owner 写入权限配置文件。
- **Workspace 路径安全**：Agent 和 User ID 拼接到路径时需防止路径穿越攻击（如 `../`）。并发创建 workspace 目录时不得产生竞态条件。
- **最小信息原则**：Agent 被拒绝时仅收到操作结果，不暴露规则名称、优先级或配置路径等内部信息。

### F9. 子 Agent 权限继承

Owner 需要子 Agent 的权限不超出父 Agent：

- 父 Agent 派生子 Agent 时，子 Agent 的实际权限 = 子 Agent 自身权限 ∩ 父 Agent 权限 ∩ 继承的 User 权限。
- 权限沿 spawn 链路只能变窄，不能变宽。
- 子 Agent 被拒绝时静默返回，不触发审批流程。

### F10. 无直接用户交互场景的降级处理

Owner 需要对定时任务等无直接用户交互的场景做权限降级处理：

- 定时任务必须在配置中指定一个 User ID。指定的 User ID 未注册时跳过任务并通知 Owner。
- 定时任务的写操作被拒绝时按配置处理：静默跳过、通知 Owner、或进入审批流程。
- 定时任务日志始终写入系统日志，供 Owner 事后查询。

## 非功能需求

- 权限检查不应成为 Agent 操作的性能瓶颈——对单次工具调用的权限判定应在 100ms 以内完成。
- 审批卡片必须在操作被拒绝后立即推送到 Owner 的 IM 渠道，延迟应在 2 秒以内。
- 权限规则变更后，下一次权限检查即使用新规则，不应出现窗口期。
- 系统应在无 Owner 在线时保持可用——等待中的审批请求不应阻塞系统其他功能。
