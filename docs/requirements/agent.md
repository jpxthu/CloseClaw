# Agent 需求

## 概述

Agent 模块定义每个 AI Agent 的身份和能力边界，让 Owner 能够为不同用途配置差异化的 Agent 实例，并通过 Agent 间的层级协作完成复杂任务。

## 功能需求

### F1. Agent 配置档案

Owner 可以为每个 Agent 定义一份独立的配置档案，包含以下能力维度：

- **身份标识**：Agent 的唯一 ID 和显示名称
- **模型选择**：Agent 使用的默认 LLM 模型及备用模型列表
- **工作目录**：Agent 的默认工作目录
- **身份加载模式**：完整模式或精简模式，控制上下文注入文件的数量
- **工具白名单/黑名单**：Agent 可以使用的工具范围
- **技能白名单**：Agent 可以使用的技能范围
- **子 Agent 控制**：Agent 创建子 Agent 的限制规则（目标白名单、层级深度、并发数、默认子 Agent 等）
- **记忆配置**：Agent 的记忆模块参数（可选覆盖默认值）

Agent 的配置档案为纯静态定义，不包含运行时可变状态。Agent 的运行时行为由 Session 模块驱动。

### F2. 身份与人格分离

Agent 的能力边界（配置档案）和身份人格（Bootstrap 文件）是两层独立定义：

- **配置档案**（JSON）定义 Agent 的能力边界——模型、工具、权限、spawn 控制
- **Bootstrap 文件**（Markdown）定义 Agent 的身份人格——操作规程、角色定义、用户偏好等

Owner 可以为不同用途定制 Agent 的身份人格文件，并在配置档案中指定身份加载模式和文件目录。身份加载模式控制上下文注入文件的数量：
- **完整模式**：加载全部身份文件（操作规程、角色定义、身份标识、用户信息、工具指南、自定义引导、长期记忆）
- **精简模式**：仅加载核心文件，减少上下文占用

### F3. Agent 能力组合

Agent 的能力由其配置字段的组合决定，不依赖预定义的类型标签。Owner 通过配置字段的排列组合来定制 Agent 的行为边界：

- 工具白名单/黑名单控制 Agent 可以执行的操作范围
- 身份加载模式控制 Agent 的上下文体积
- 子 Agent 控制参数决定 Agent 的繁衍能力
- 权限配置决定 Agent 的安全边界

框架提供一组预置的行为模板（如"只读研究"、"校验审计"），在创建子 Agent 时可选择注入对应的行为约束，但不影响 Agent 本身的配置定义。

### F4. 配置层级与优先级

Agent 配置支持项目级和用户级两层配置，采用字段级覆盖合并：

- **项目级配置**（`<项目>/.closeclaw/agents/<id>/`）优先级最高
- **用户级配置**（`~/.closeclaw/agents/<id>/`）次优先级
- 高优先级配置中未指定的字段回退到低优先级配置的值
- 合并后生成完整配置，所有字段均有确定值，不存在空值

Agent 通过注册清单显式管理：只有清单中列出的 Agent ID 才会被加载，不在清单中的配置目录即使存在也忽略。被注释掉的 ID 视为未注册。

### F5. 初始 Agent 创建

Owner 首次使用系统时，CLI 配置向导引导创建一个初始 Agent。初始 Agent 的 ID 默认为 `master`，具备全部工具和技能的访问权限，作为系统的基础入口。

### F6. 运行时配置查询

系统运行时，各模块通过 Agent ID 查询 Agent 的完整配置。查询是只读的，返回该 Agent 的完整能力定义（模型、工具集、技能列表、子 Agent 控制参数等）。

配置支持热重载：Owner 修改配置文件后，系统检测到变更并更新内存中的配置数据。已运行的会话是否感知变更由各消费模块自行决定。

### F7. 子 Agent 创建（Spawn）

Agent 可以创建子 Agent 来执行子任务。创建子 Agent 时，调用方指定：

- **目标 Agent**：使用哪个 Agent 配置（默认使用父 Agent 配置中的默认子 Agent）
- **任务描述**：子 Agent 要完成的任务
- **运行模式**：一次性执行（完成后自动结束并回传结果）或持久存活（等待后续控制）
- **上下文模式**：子 Agent 仅接收任务描述，还是继承父 Agent 的对话历史
- **上下文精简**：子 Agent 是否以精简身份加载模式启动
- **模型覆盖**：可选覆盖目标 Agent 的默认模型
- **行为模板**：可选注入预置的行为约束模板（如"只读研究"、"校验审计"）
- **工作目录覆盖**：可选为子 Agent 指定独立的工作目录

子 Agent 的最终模型按以下优先级确定：显式指定的模型 > 父 Agent 配置中的子 Agent 默认模型 > 目标 Agent 自身配置的模型 > 系统默认模型。

### F8. 子 Agent 上下文继承（Fork）

Fork 模式是 Spawn 的一种变体：子 Agent 在创建时继承父 Agent 的完整对话历史，使子 Agent 理解已发生的上下文后再执行新任务。与普通 Spawn 的区别：普通 Spawn 的子 Agent 只看到任务描述，Fork 模式的子 Agent 先看到父 Agent 的对话历史，再看到任务描述。

### F9. Spawn 创建控制

Owner 可以通过父 Agent 的配置控制子 Agent 的创建行为：

- **目标白名单**：限制可以 spawn 的目标 Agent 范围（泛配 `*` 表示不限制，空列表表示禁止 spawn）
- **层级深度**：限制最多可以嵌套多少层子 Agent（0 表示禁止 spawn 任何子 Agent）
- **并发数量**：限制同时存活的子 Agent 数量上限
- **必选 Agent ID**：要求 spawn 时必须显式指定目标 Agent
- **默认子 Agent**：spawn 不指定目标时的默认值

层级深度受父 Agent 和自身配置的双重约束：取两者中更严格的值生效。即使父 Agent 允许更多层级，子 Agent 可以主动收窄自己的子树深度。

### F10. 子 Agent 结果回传

一次性执行的子 Agent 完成后，执行结果通过推送机制自动回传给父 Agent。父 Agent 在下一次对话轮次开始时处理子 Agent 的结果。

父 Agent 不需要主动轮询子 Agent 的状态。子 Agent 完成后自动推送，父 Agent 在下一轮对话中自然地获取结果。

### F11. 持久子 Agent 控制

持久存活的子 Agent 允许父 Agent 进行运行时控制：

- **Steer**：向存活的子 Agent 发送新任务，子 Agent 重新执行
- **Kill**：终止子 Agent 及其所有后代子 Agent，释放运行资源。子 Agent 的对话历史保留，Owner 仍可查阅
- Kill 采用级联清理：从最深层向浅层逐级终止，确保所有后代都被清理，不留下孤立节点

以下场景也触发自动级联清理：
- 父 Agent 会话正常结束时，所有仍活跃的子 Agent 被自动终止
- 父 Agent 会话超时被清理时，所有子 Agent 被级联终止

### F12. Agent 权限继承

子 Agent 的实际权限沿创建链路收窄：

- 子 Agent 的最终权限 = 子 Agent 自身配置权限 ∩ 创建链路中所有父 Agent 的实际权限 ∩ 当前用户权限
- 权限只能收窄，不能放宽：父 Agent 被禁止的操作，子 Agent 也一定被禁止
- 子 Agent 权限被拒绝时，直接返回权限拒绝错误，不进入用户审批流程（子 Agent 不是面向用户的入口）
- Owner（系统最高权限身份）在权限评估中跳过用户维度，仅受 Agent 维度约束

父 Agent 配置中的拒绝规则沿链路向子 Agent 传播，构成额外的拒绝约束。每增加一级深度，拒绝约束只增不减。权限评估每次新鲜计算，不缓存结果，确保权限变更即时生效。

### F13. 工作目录权限

每个 Agent 在其自身的工作目录内自动获得读写权限。工作目录由以下顺序确定：
- spawn 时显式指定的路径
- 目标 Agent 配置中的工作目录
- 以上均未指定时，使用父 Agent 工作目录下的子目录

工作目录权限在操作权限评估之前生效——即使 Agent 的权限规则未覆盖该路径，工作目录内的文件仍可读写。

### F14. Agent 间通信

子 Agent 创建时，系统自动配置父子之间的消息路由：

- 子 Agent 默认可以向父 Agent 发送消息
- 父 Agent 默认可以接收子 Agent 的消息
- Agent 间的消息送达需要同时满足路由配置和权限允许两个条件

通信路由在 spawn 时自动生成，默认仅父 Agent 可达。如需扩展到其他 Agent，需要额外配置。

### F15. 权限热更新

Owner 修改 Agent 权限配置后，系统在下一次权限评估时使用新规则，不缓存旧的评估结果。修改权限配置不影响 Agent 核心配置，反之亦然。权限文件不存在时不阻塞 Agent 加载，使用系统默认权限。

### F16. Spawn 层级追踪

系统维护 Agent 之间的创建层级关系，记录每个会话的父会话标识、目标 Agent ID、当前层级和运行模式。层级关系支持以下查询：
- 查询某个会话的所有直接子会话
- 查询某个会话的子树（所有后代）
- 查询某个会话的父会话

系统重启后，从持久化的会话数据中恢复层级关系。如果父会话已不存在（已被清理），子会话降级为独立根节点继续服务，不丢失数据。

## 非功能需求

### 性能

- Agent 配置查询为纯内存操作，单次查询延迟不影响会话创建
- 权限评估每次新鲜计算，不缓存结果，确保任何权限变更即时生效
- 子 Agent 结果回传采用推送机制，不需要父 Agent 轮询

### 安全性

- 子 Agent 的权限沿创建链路只收窄不放宽
- 子 Agent 被拒绝时不进入用户审批流程，避免非用户入口的权限提升
- 权限文件和配置文件独立存储、独立热更新，互不影响

### 可用性

- Agent 配置文件缺失时不影响系统运行，使用系统默认值
- 权限文件缺失时不阻塞 Agent 加载，使用系统默认权限
- 系统重启后自动恢复 Agent 层级关系，已丢失父会话的子会话降级为独立会话继续服务

### 可扩展性

- Agent 能力由配置字段组合决定，不依赖预定义的类型枚举
- 新增能力维度只需新增配置字段，不需要修改 Agent 类型定义
- 项目级和用户级两层配置支持团队协作和个人定制的场景
