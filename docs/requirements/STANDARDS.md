# 需求文档写作标准

## 定位

需求文档纯粹记录用户（owner）的需求、使用场景和日常使用习惯。不讨论技术实现、不涉及架构设计。

**信息来源**：需求文档的信息**只来源于 owner 本人**，以及和 owner 一起进行的调研和商讨结论。不得从代码、设计文档或其他来源自行推断需求。

**与设计文档的关系**：设计文档是需求和代码之间的桥梁——将需求翻译为实现层的架构蓝图。设计文档**必须遵从需求文档**，若设计文档的方案与需求文档冲突，以需求文档为准，设计争议提给 owner 裁决。

**关联方向**：设计文档的 README 中必须引用对应的需求文档。需求文档的「关联设计文档」节仅供快速反向查找。

## 术语

- **Owner**：项目所有者，唯一的产品决策者
- **User**：CloseClaw 的最终使用者，含 Owner 和非 Owner User
- **Agent**：AI Agent，系统的另一类使用者（通过工具和 prompt 与系统交互）

## 文件组织

```
docs/requirements/
├── README.md              ← 需求文档总索引
└── <模块名>.md            ← 每个模块一个需求文件
```

规则：
- 一个模块一个文件，不拆子功能文件
- 模块名与 `docs/design/` 目录对应
- 需求跨模块时，放在最核心的模块文件中，其他模块引用

## 章节结构

固定 6 节，不增不减。

```
# <模块名> 需求

## 概述
一句话：这个模块要满足用户的什么核心诉求

## 用户场景
每条格式：作为<角色>，我想<做什么>，以便<获得什么价值>
覆盖模块涉及的所有角色视角（如 Owner、非Owner User、Agent 等）

每条场景前标注优先级：
- 【P0】— MVP 必须有，缺了产品不可用
- 【P1】— v1.1 上线
- 【P2】— 后续迭代

## 验收标准
每条用户场景对应 3-5 条 Given/When/Then 可测断言
验收标准可作为 e2e 测试用例的来源

## 用户期望行为
描述用户看到什么、操作什么、系统如何响应
- 正常流程：用户操作 → 系统响应
- 异常流程：错误场景 → 系统提示与兜底
- 边界条件：极端输入 / 断网 / 磁盘满 / 权限撤销等场景的用户感知

UX 细节级别：写到文案、按钮标签、卡片字段结构、通知格式。
不需要像素级 UI 稿，但必须能据此验收"用户交互是否符合预期"。

## 关联设计文档
对应 `docs/design/` 下的设计文档位置，仅供快速反向查找。
设计文档 → 需求文档 的引用关系是权威方向，设计文档 README 中维护。

## 非功能需求
性能、可用性、安全性、可观测性等质量属性的用户期望。
以用户可感知的指标表述（如"审批卡片必须在 2 秒内推送到 Owner"），
不写技术指标（如"数据库连接池大小"）。
```

## 红线

**不写**：
- 技术实现方案、架构设计、模块接口
- 代码相关内容（函数名、变量名、crate 名、数据结构）
- 设计取舍过程、技术选型讨论
- 现状问题（"当前代码中……"）
- 借鉴来源（"参考 XXX 产品"）
- 未来计划、开发步骤
- 超出 6 节的内容
