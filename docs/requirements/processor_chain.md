# Processor Chain 需求

## 概述

所有进出 CloseClaw 的消息需要经过统一的处理管道——入站标准化文本、计算追踪标识，出站过滤冗余内容、解析交互指令、记录审计日志。

## 功能需求

### F1. 入站消息统一处理

所有 IM 平台的消息在进入系统核心逻辑前，必须经过统一的入站处理流程。处理流程按固定顺序执行以下操作：原始消息日志记录、会话路由标识计算、文本内容标准化。

处理流程对上层透明——上层模块接收到的是处理完成的消息，无需关心平台差异或处理细节。

### F2. 会话路由标识计算

每条入站消息都需要生成唯一的会话路由标识（session_key），用于消息追踪和调试。标识由时间戳和路由字段（平台、发送者、会话对端、账号标识）的哈希值组合而成，毫秒级时间戳天然防撞。

路由计算失败时不应阻塞消息流转，但需记录告警日志。

### F3. 文本内容标准化

入站消息的文本内容在送达 LLM 之前需要做标准化处理：去除控制字符和 ANSI 转义序列、压缩连续空行、去除行尾空格。非文本消息（图片、文件、音频）跳过标准化直接透传。

标准化失败时回退到原始内容，不丢失消息。

### F4. 出站回复冗余控制

LLM 生成回复后、发送到 IM 平台之前，Owner 能按层级控制回复包含的细节量：
- **完整**：包含所有内容块（文本、思维链、工具调用结果）
- **标准**：移除思维链块，保留文本和工具结果
- **仅文本**：只保留纯文本块

冗余控制在 DSL 指令解析之前执行，确保被过滤掉的内容不参与后续处理。

### F5. DSL 交互指令

Agent 的回复文本中可嵌入简单的 DSL 指令来生成交互式 UI 元素（按钮、选择器等）。DSL 指令在出站处理中自动识别、解析并从可见文本中移除，解析结果随消息传递到渲染层。

Owner 可以在 system prompt 或 bootstrap 中教会 Agent 使用预定义的 DSL 指令来生成交互元素。

### F6. 消息审计日志

入站原始消息和出站处理后的消息均可写入审计日志文件，按配置开关。日志用于故障排查、消息回溯和行为审计。日志写入失败不影响消息正常流转。

出站日志仅在实际配置日志目录时生效，未配置时不记录。

## 非功能需求

1. **处理不可阻塞消息流**：任何处理步骤异常时，必须回退到透传原始内容，消息继续流转。不可因单个步骤失败导致消息丢失。
2. **低延迟**：入站和出站处理不应成为消息链路中的感知瓶颈，处理总耗时应在毫秒级。
3. **可扩展**：新增处理器时不需修改已有处理器或框架代码，只需注册新的处理步骤和优先级。
4. **可观测**：所有处理异常必须记录日志（含足够上下文用于定位），告警级异常需区分于普通日志。
