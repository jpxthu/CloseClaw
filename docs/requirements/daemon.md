# Daemon 需求

## 概述

Owner 需要一个可靠的守护进程：系统启动时所有组件按正确依赖顺序初始化完成后才对外提供服务，关闭时安全持久化所有数据并通知进度，异常中断后衔接恢复流程。

## 功能需求

### F1. 系统启动

Owner 需要系统启动全流程可控：从进程启动到所有组件就绪，组件只在依赖就绪后才被初始化。所有外部入口（IM 渠道、CLI 等）就绪后，系统才开始接收和处理消息。

### F2. 优雅关闭

Owner 需要通过系统信号触发优雅关闭：

- 关闭一旦开始，系统不再接收新消息
- 正在进行的会话允许完成当前工作：LLM 流式输出直到流结束，工具执行直到执行完毕
- 未开始的新 LLM 轮次不再触发，下次用户消息到来后按会话上下文继续
- 所有会话数据在关闭前持久化到存储

Owner 可以在优雅关闭等待期间随时升级为强制关闭（方式详见 F3）。

### F3. 强制关闭

Owner 需要通过重复信号或在关闭进度通知中选择"强制关闭"来立即终止系统：

- 正在运行的工具进程被终止
- 正在进行的 LLM 请求被取消，当前未完成的响应片段丢弃
- 因跳过 F2 的持久化流程，未持久化的会话数据在关闭最后阶段兜底持久化
- Owner 接受强制关闭带来的工具副作用（如编译到一半的文件状态不可控）

系统下次启动时由 session 模块的恢复机制处理强制关闭遗留的未完成操作。

### F4. 关闭进度可见性

Owner 需要在优雅关闭期间看到实时进度：哪些会话还在活跃、各自当前状态（LLM 输出中/工具执行中/已就绪）、已等待时长。Owner 可以通过进度通知选择"继续等待"或"强制关闭"。

进度信息在会话状态变化时更新。大段等待无变化时也应有周期性更新，让 Owner 确认系统仍在关闭中而非卡死。

### F5. 后台任务生命周期

Owner 需要系统的后台定时任务在启动时自动开始运行、关闭时有序停止：正在执行的任务应有机会自行结束；未及时完成的任务被终止后不应留下副作用。所有后台任务停止后不残留未完成的工作。

## 非功能需求

- 系统从进程启动到接收第一条消息的初始化耗时不应让 Owner 感到等待过长
- 优雅关闭不应无限等待——Owner 需要感知长耗时会话的存在并自主决定是否继续等待
- 强制关闭后的恢复不应丢失已持久化的会话数据
- 关闭进度通知应在关闭流程开始后立即推送给 Owner，不应有可感知的延迟
