# LLM 模块需求

## 概述

用户可以使用多家 AI 供应商的模型进行对话和工具调用，无需关心底层协议差异，获得一致的聊天体验。

## 功能需求

### F1. 多供应商统一对话

User 或 Agent 可以使用不同 AI 供应商的模型发起对话。支持的供应商包括 MiniMax、GLM（智谱）、DeepSeek、火山方舟（VolcEngine）、MiMo（小米）。

User 可以查询供应商账户的余额（如 DeepSeek）或用量配额（如 GLM 的套餐等级、时间窗口限额和已用百分比）。

对话过程中，系统维护完整的上下文——每轮对话的消息追加到历史中，不做修改和删除。

User 收到的响应内容在语义上是统一的：文本回复、推理/思考过程、工具调用请求、工具执行结果。User 不需要知道底层是 OpenAI 协议还是 Anthropic 协议。

### F2. 流式实时输出

User 可以在 LLM 逐 token 生成响应时实时看到内容，无需等待完整响应。

流式输出过程中如果发生错误，已输出的文本片段保留给 User，同时向 User 展示错误提示。不完整的回复不写入对话上下文。

### F3. 推理过程可见与安全

User 可以看到模型的推理/思考过程。推理内容作为独立内容块保留在对话历史中，参与后续对话的上下文。

默认情况下，主终端仅展示推理进行中的状态指示，不展示完整推理文本。User 可通过详情面板按需查看。

推理内容不会在流式输出中被泄露——流式输出只向目标 User 展示文本回复，不展示推理过程。

### F4. 推理强度控制

User 可以调整模型的推理强度档位（low / medium / high / max），默认 high。

User 可以通过运行时指令查询当前会话的推理强度档位，也可以修改当前会话的档位（修改仅对当前会话生效，不持久化到配置）。

User 可以请求关闭推理。取决于供应商能力——部分供应商的模型设计上总是输出推理内容（如 DeepSeek 的 thinking 无法真正关闭，MiMo 在所有场景下均输出推理），关闭请求在这些供应商上仅将推理强度降至最低档位。

当 User 输入非法档位值时，系统不报错，回显当前生效的档位。

当所选模型不支持所请求的档位时，系统自动降级到该模型支持的最高档位，不打扰 User。查询指令会显示实际生效值。

### F5. 工具调用

Agent 可以在对话中调用工具——系统向 AI 模型发起请求后，模型可以发出工具调用指令，系统执行后将结果返回给模型。

在工具调用的多轮交互中，上一轮的推理内容在下一轮请求中需带回给模型，确保上下文连续性。

### F6. 供应商配置向导

User 可以通过 CLI 交互式向导完成 LLM 供应商的初始配置，无需手动编辑配置文件。向导流程为：选择供应商 → 输入凭据 → 发现可用模型 → 选择模型 → 确认并写入配置。

凭据输入时不回显字符，输入完成后给出即时反馈告知 User 凭据已接收。凭据独立于配置文件存储，配置文件只引用凭据路径，不存明文。

向导可重复运行。若已有供应商和凭据，再次运行时自动跳过供应商选择和凭据输入步骤，直接从模型发现步骤开始。再次运行时已配置的模型默认预选。User 取消选择某个模型后，该模型不会从已有配置中删除。

模型选择支持多选、范围语法（如 1-3,5）和全选关键字。

向导中 User 可以随时取消，不留下中间状态文件。

### F7. 模型发现

系统在配置过程中可以自动从供应商 API 获取可用模型列表，结合内置知识库自动填充每个模型的推荐参数（上下文窗口大小、最大输出 token 数、默认温度、是否为推理模型、推荐协议、支持的输入类型等）。

动态探测优先——先调用供应商 API 获取最新列表。API 不可用时回退到内置知识库，并向 User 标注回退来源。

系统仅支持知识库中已知的模型，API 返回的未知模型不会被展示。

### F8. 缓存成本优化

系统利用供应商的缓存机制降低 API Token 费用。对话中不变的内容（系统提示中的引导信息、工具定义等）被缓存到 AI 服务端，后续请求命中缓存时仅收取少量费用。

缓存策略按供应商适配：支持显式缓存标记的供应商（如 Anthropic、MiniMax）由系统自动标记缓存控制参数；支持会话键缓存注入的供应商（如 Kimi）由系统传入会话标识帮助服务端关联缓存；依赖服务端自动前缀缓存的供应商（如 DeepSeek、OpenAI）由系统保证请求前缀稳定性来维持缓存命中。

### F9. 用量统计

User 可以看到每次 API 调用的 Token 用量，包括缓存命中与缓存写入的统计，以及推理消耗的 token 数（reasoning_tokens，与文本输出 token 分开统计）。

跨多轮对话累积缓存统计，结束会话时清零。如果 API 响应不携带缓存字段，对应字段显示为 0。

## 非功能需求

### 性能

- 流式输出的首 token 延迟应尽量低，User 感知到"模型在思考"后尽快看到第一个输出 token
- 模型发现 API 探测应在 10 秒内完成，超时后自动回退到内置知识库
- 缓存命中率应被持续监控，静态区内容的前缀缓存在连续对话中应持续命中

### 可用性

- 配置向导中凭据输入失败时，给出明确的错误原因（认证失败 / 网络超时 / 权限不足），允许 User 重试或跳过
- 模型发现失败时自动回退到内置知识库，不阻塞配置流程
- 流式输出中断时，已输出的内容不丢失，向 User 展示错误提示

### 可靠性

- 系统为每个模型选择适用的协议，不应因协议错配导致目标能力（如 thinking、工具调用）静默丢失
- User 请求关闭推理时，供应商不支持关闭的不视为错误——系统降至最低可用档位

### 安全性

- 供应商 API 凭据独立于配置文件存储，配置文件不包含明文凭据
- 推理内容不通过流式输出泄露给非目标 User
- 凭据输入时不回显字符

### 可观测性

- 记录每次 API 调用的 Token 用量和缓存命中情况，User 可查询累积统计
- API 调用失败时记录错误原因和上下文，便于排查
