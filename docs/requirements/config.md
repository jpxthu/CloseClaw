# Config 需求

## 概述

配置模块满足 Owner 对 CloseClaw 配置的日常管理需求：单文件损坏不影响整体运行、写入操作不会损坏已有配置、敏感凭据与业务配置安全隔离。配置的生效时机由各消费模块决定，配置模块提供变更检测和通知能力。

## 功能需求

### F1. 多文件配置结构

Owner 的 CloseClaw 配置按职责拆分为多个独立配置文件，存放在 `~/.closeclaw/config/` 目录下。一个文件损坏不影响其他文件的加载。

配置文件按职责划分，包括模型定义、渠道配置、账户绑定、Gateway 参数、插件列表、会话参数、系统设置等。

CloseClaw 通过显式注册清单管理 Agent 加载：只有在注册清单中列出的 Agent ID 才会被加载，目录中存在但未注册的 Agent 配置会被忽略。注册清单支持注释格式，注释掉某行即取消注册。Agent 的独立配置文件存放在 `~/.closeclaw/agents/<agent-id>/` 目录下。

项目仓库可以包含项目级 Agent 注册清单和配置，与用户级配置合并使用：Agent ID 列表取并集，配置字段以项目级优先覆盖用户级。

### F2. 安全写入

系统修改配置文件时，校验失败或写入中断不会产生部分覆盖或损坏的配置文件，原有配置保持不变。

### F3. 启动容错与备份恢复

CloseClaw 启动时，若某个配置文件损坏或无法解析，自动使用最新的一份备份恢复。备份存放在配置目录的 `.backups/` 子目录下，每个配置文件保留最近若干份备份。

恢复成功后系统正常启动并记录警告；系统恢复失败（如无可用备份）时拒绝启动。配置文件存在但部分字段缺失时，缺失字段使用默认值加载，不报错、不阻塞启动。凭据文件加载失败的处理见 F5。

### F4. 配置重载

Owner 在 CloseClaw 运行时编辑配置文件后，变更的生效时机取决于各消费模块的设计，不由配置模块统一决定。部分模块加载最新文件即生效（如权限模块），部分模块在新会话创建时生效（如会话参数），部分核心配置需要重启 Gateway 才能生效。以上三类为典型情形，实际以各消费模块的设计为准。

配置模块检测到文件变更后通知相关模块，各模块自行决定重载策略。对同一文件短时间内发生的多次写入合并为一次重载，避免每次保存都触发重载。热重载校验失败时，保留内存中的旧配置继续运行，不恢复备份文件，通过 IM 推送通知 Owner。若该配置在内存中无旧值（首次加载后即损坏），则阻塞该配置的加载及依赖该配置的后续流程。

### F5. 凭据安全隔离

API Key 等敏感凭据文件与业务配置文件物理隔离，存放在独立的 `credentials/` 子目录下，按服务商分文件存储。业务配置只引用服务商名称，凭据在系统读取配置时自动关联。凭据文件加载失败不阻塞系统启动，仅影响依赖该凭据的功能。

### F6. 配置版本兼容

CloseClaw 配置结构变更时，已有配置文件必须可被新版本正常加载。旧配置中缺失的新字段使用默认值，不再支持的旧字段被静默忽略，不阻塞启动。
