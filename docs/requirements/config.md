# Config 需求

## 概述

配置模块满足 Owner 对 CloseClaw 配置的日常管理需求：修改配置后即时生效无需重启、单文件损坏不影响整体运行、敏感凭据与业务配置安全隔离、写入操作不会损坏已有配置。

## 功能需求

### F1. 多文件配置结构

Owner 的 CloseClaw 配置按职责拆分为多个独立配置文件，存放在 `~/.openclaw/config/` 目录下。一个文件损坏不影响其他文件的加载。

配置文件按职责划分，包括模型定义、渠道配置、账户绑定、Gateway 参数、插件列表、会话参数、系统设置、Agent 注册清单等。Agent 的独立配置（`config.json`、`permissions.json`）存放在 `agents/<agent-id>/` 目录下。

Agent 通过显式注册清单管理：只有在注册清单中列出的 Agent ID 才会被加载，目录中存在但未注册的 Agent 配置会被忽略。注册清单支持 JSONC 格式，注释掉某行即取消注册。

项目仓库可以包含项目级 Agent 配置与注册清单，与用户级取并集后合并使用，字段级覆盖。

### F2. 安全写入

系统通过统一的写入入口修改配置文件。每次写入遵循：先校验新内容 → 备份当前文件 → 原子写入新内容 → 更新内存缓存。任一步失败即中止，不产生部分写入或损坏的配置文件。

### F3. 启动容错与备份恢复

CloseClaw 启动时，若某个配置文件损坏或无法解析，自动使用最近的备份文件恢复。备份存放在配置目录的 `.backups/` 子目录下，每个配置文件保留最近若干份历史备份。

恢复成功后系统正常启动并记录警告；恢复失败（无可用备份）则拒绝启动。配置文件存在但部分字段缺失时，缺失字段使用默认值加载，不报错、不阻塞启动。凭据文件损坏不阻塞启动，仅影响需要该凭据的功能。

### F4. 配置热重载

Owner 在 CloseClaw 运行时直接编辑配置文件后，变更自动生效，无需重启 daemon 或中断已有会话。系统监听配置文件变更，仅重新加载发生变更的文件，校验通过后推送到内存和已存在会话。新会话直接使用最新配置。

短时间内的多次变更合并处理，避免频繁重载。

### F5. 凭据安全隔离

API Key 等敏感凭据与业务配置文件物理隔离，存放在独立的 `credentials/` 子目录下，按服务商分文件存储。业务配置只引用服务商名称，凭据在加载时动态注入。凭据文件加载失败不影响系统启动，仅影响需要该凭据的功能。

### F6. 配置迁移

Owner 从旧版本（单文件 openclaw.json）升级时，系统自动检测并引导迁移：将单文件配置拆分为 config/ 目录下的多文件结构，完成后保留原文件备份。已迁移到多文件结构的系统直接从 config/ 加载，不再依赖旧单文件。

## 非功能需求

- 配置文件变更后，已有会话在秒级内感知新配置并生效
- 配置写入操作的校验阶段在毫秒级完成，不阻塞调用方
- 凭据文件依靠文件系统权限保护，权限仅限于 Owner 可读写
