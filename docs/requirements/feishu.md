# 飞书 需求

## 概述

飞书平台需求覆盖 CloseClaw 与飞书的完整交互能力——用户通过飞书私聊与 Agent 对话、通过飞书群聊与 Agent 协作，Agent 输出的富格式内容（卡片、按钮、代码块等）在飞书中正确渲染，同时 Agent 能够使用飞书平台特有工具（IM 操作、日历、文档等）。

## 功能需求

### F1. 飞书入站消息接收

用户通过飞书发送的消息需被正确接收和解析。支持以下消息类型：

- **文本消息（text）**：提取消息正文文本
- **富文本消息（post）**：展开富文本结构为可读文本，包含标题、段落、列表（有序/无序）、文本样式（粗体、斜体、删除线、下划线、行内代码、引用块）以及结构元素（@提及、链接、邮箱、电话）的语义保留。内嵌的媒体内容以占位符表示
- **引用/回复消息**：提取被引用消息的文本内容（截断至 500 字符），拼接在本消息正文前，以引用格式呈现

话题消息中的话题 ID 需被提取，用于后续回复时定位到正确的话题。

非文本消息（图片、文件、语音）：当前不进行语义理解处理。

### F2. 飞书消息发送

Agent 的回复需通过飞书发送给用户。支持两种消息类型：

- **文本消息**：纯文本、无格式标记的单行内容，以飞书文本消息接口发送
- **卡片消息**：含格式（标题、粗体、斜体、代码块、列表、引用、链接、分割线、换行）或含交互按钮或多内容块的内容，以飞书交互式卡片消息接口发送

消息格式选择自动完成，用户无需关心底层发送方式。

若消息来自群聊中的话题，回复需定位到原话题下（以回复形式发送）。

### F3. 飞书卡片渲染

Agent 输出的富格式内容需正确渲染为飞书卡片。

**标题提取**：首行 `#` 标题自动提取为卡片标题（蓝色模板），标题行不进入正文。

**正文元素映射**：所有标准 markdown 元素需对应到飞书卡片原生元素：

| 内容 | 飞书卡片呈现 |
|------|------------|
| `# 标题` | 卡片标题（蓝色） |
| `##` 及更低级标题 | 原生标题样式 |
| 粗体 / 斜体 | 原生粗体 / 斜体 |
| 行内代码 | 行内代码样式 |
| 代码块 | 原生代码块（平台自行高亮） |
| 引用 | 引用块 |
| 列表（有序/无序） | 原生列表 |
| 链接 | 可点击链接 |
| 分割线 | 水平分割线 |

**特殊块渲染**：
- **Thinking 块**：以折叠推理区块的形式展示，与正文可区分
- **Tool 调用块**：以工具调用描述的形式展示，包含工具名和参数概要
- **Tool 结果块**：以工具结果内容的形式展示

### F4. 飞书交互按钮

Agent 输出中的交互指令（按钮）需在飞书卡片中渲染为可点击按钮组件。

- 首个按钮渲染为主要样式按钮（突出显示）
- 后续按钮渲染为默认样式按钮
- 所有按钮平铺在卡片底部的操作区域

按钮携带操作指令，用户点击后系统应能识别按钮动作并执行对应操作。

### F5. 飞书平台工具

Agent 应能使用飞书平台的以下工具能力，在对话中完成平台特有操作：

| 工具分组 | 能力范围 |
|---------|---------|
| 飞书 IM 操作 | 发送消息、撤回消息、编辑消息、表情回应等 |
| 飞书日历 | 日程创建、查询、修改、删除 |
| 飞书任务 | 任务创建、查询、修改、删除 |
| 飞书多维表格 | 表格数据读写、记录管理 |
| 飞书文档 | 文档创建、编辑、读取 |
| 飞书云盘 | 文件上传、下载、管理 |
| 飞书电子表格 | 表格创建、编辑、读取 |

所有工具在首次被 Agent 调用时才初始化，避免启动时不必要的加载。

### F6. 飞书话题回复

群聊中话题消息的回复需正确关联到原话题。

入站时解析话题 ID，出站时携带话题 ID 定向回复。话题 ID 不参与会话路由（会话按群聊粒度管理），仅用于消息发送时的回复定位。

## 非功能需求

### 性能

- 卡片消息渲染不应显著增加消息发送延迟（用户感知发送到看到的时间与普通 IM 消息相当）

### 可靠性

- 飞书 API 调用失败时，系统应正常降级，Agent 继续运行不受影响
- 工具调用失败时，Agent 应收到清晰的错误反馈，以便做出合理响应

### 可观测性

- 非文本入站消息应记录日志，便于后续分析是否需要扩展支持
