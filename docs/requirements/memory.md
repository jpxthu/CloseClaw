# Memory 需求

> **实验性功能**：Memory 模块为实验性功能。已涵盖大框架和整体思路，其中的标签体系、记忆提取方式、记忆库搜索方式等细节处于实验迭代中。

## 概述

Agent 在跨会话的使用中积累对用户的认知——记住用户纠正过什么、做过什么决策、Agent 自己发现了什么规律，在后续对话中自动引用这些记忆，无需用户重复说明。

记忆体系围绕两个核心概念组织：

- **事件（event）**：从会话中挖掘出的值得长期保留的信息单元。每条事件包含标题、摘要、正文、类别（error / anger / decision / insight）、可选的经验教训（lesson），并关联若干实体。
- **实体（entity）**：事件关联的对象——可以是人、主题、产品、动作等。每个实体指定一种类型、名称和描述。实体类型体系沿用 [SAG](https://github.com/Zleap-AI/SAG)（MIT License）的 11 种类型定义。

## 功能需求

### F1. 会话结束后自动挖掘记忆

每次会话结束后，系统自动从对话记录中挖掘值得长期保留的信息，不依赖 Agent 在对话中自行记录。

- 挖掘过程以独立会话运行，不与用户交互
- 挖掘产出结构化的事件（event），每条事件包含：标题、摘要、正文、类别、可选的经验教训（lesson）
- 事件分四类：
  - **错误（error）**：Agent 在对话中出现的明确错误（判断失误、操作不当、误解需求）
  - **不满（anger）**：用户对 Agent 的纠正或不满表达
  - **决策（decision）**：用户明确的偏好选择或设计决策
  - **洞察（insight）**：Agent 经过反复尝试发现的简单规律，值得沉淀为经验的发现
- 从错误和不满事件中提炼可执行的经验教训——简洁的行为指导，不引用具体消息编号
- 挖掘只提取对未来对话有参考价值的持久信息，不记录临时聊天细节（如一次性闲聊、当前任务进度等）
- 挖掘时参考已记录的事件和已有规则，避免重复产出
- 对话轮数或用户消息数过少的简短会话自动跳过，不挖掘
- 每次挖掘产出的事件数有上限，优先保留高信号事件（错误和不满优先于普通决策）

### F2. 以实体体系组织记忆

记忆围绕实体（entity）组织——把事件关联到具体的人、主题、产品、动作等，便于后续搜索和关联。实体类型体系沿用 [SAG](https://github.com/Zleap-AI/SAG)（MIT License）。

- 系统提供 11 种实体类型：时间（time）、地点（location）、人物（person）、组织（organization）、主题（subject）、产品（product）、指标（metric）、动作（action）、作品（work）、群体（group），以及作为兜底类型的标签（tags，暂定）
- 不同类型的实体在记忆搜索和升格中的重要性不同——主题和动作是最高价值类型
- 挖掘时为每条事件分配关联实体：优先匹配已有实体，无匹配时创建新实体
- 同一 Agent 下同类型同名实体自动合并，不重复创建
- 实体名称简洁（不超过 10 个单词）
- 不同 Agent 的实体相互隔离

### F3. 定期将事件升格为行为规则

系统定期消费已挖掘的事件，通过评分和浓缩，将反复出现的高价值经验升级为持久的行为规则。

- 升格过程自动运行，按可配置的时间周期触发
- 升格评分综合考量以下维度：
  - 事件关联的实体跨会话出现的频次（出现越频繁，评分越高）
  - 事件的新旧程度（越新的事件评分越高）
  - 用户表述的明确程度（用户明确表述比 Agent 推断评分更高）
  - 同一实体概念在多个 Agent 的会话中出现的程度（跨 Agent 同名同类型实体可对比，加分项）
- 评分不足的事件不进入浓缩环节
- 行为规则产出上限可配置，超出上限时保留高分条目
- 浓缩产出的规则直接可执行——Agent 可以直接理解并遵循
- 升格后的规则写入用户可查阅的规则文件（见 F5），原始事件保留可供后续搜索和升格使用
- 升格只产出规则，不影响原始事件——下一轮升格不会把自己的产出当作新的待升格事件

### F4. 对话中自动注入相关记忆

每次用户或 Agent 发送消息时，系统自动搜索与该消息相关的记忆，浓缩后注入对话，无需 Agent 自行判断是否需要搜索。

- 搜索从当前消息和最近对话上下文中提取查询概念
- 在已有实体关联体系中匹配相关实体，再通过实体反查关联事件
- 搜索到的相关事件浓缩为简短摘要后注入消息列表
- 同一会话中已注入过的记忆不重复注入
- 搜索结果注入位置因触发者而异——用户消息触发时置于用户消息之后，Agent 消息触发时置于下一轮用户消息之前
- 搜索超时则放弃本轮，不阻塞对话
- 挖掘和升格自身的会话不触发搜索——这两类会话不需要消费自己的产出

### F5. 用户可查阅的行为规则

升格产出的行为规则以可读文件形式持久化，用户可以直接查看和手动编辑。

- 规则文件清晰列出当前生效的所有行为规则
- 用户手动编辑后，Agent 在下次注入时使用最新内容
- 规则文件是 Agent 的 system prompt 中记忆部分的唯一来源——日常搜索产出的浓缩摘要不进入 system prompt

> **交叉引用**：规则文件注入 system prompt 的机制详见 [system_prompt §F3](system_prompt.md)。

### F6. 可选的叙事摘要

升格完成后，系统可以生成一份自然语言叙事摘要，描述本轮从事件到规则的升格过程。

- 叙事摘要仅供用户查阅，不参与记忆升格或注入链路
- 叙事摘要功能可独立开关

### F7. 记忆功能的精细化开关

用户可以为记忆体系的三个子功能（挖掘、升格、搜索）分别控制开关。

- 未显式开启则全部关闭——记忆系统不会在用户不知情的情况下运行
- 所有开关支持全局配置和按 Agent 分别配置
- 按 Agent 配置时：Agent 不声明则继承全局配置；Agent 声明部分字段则仅覆盖其声明的字段
- 配置修改后无需重启，下次触发时自动生效
- 各子功能使用的 LLM 模型可独立配置，不强制使用主对话模型
- 搜索超时时间、上下文轮数、摘要长度、事件产出上限等参数均可按需调整

> **交叉引用**：配置体系的目录结构和热重载机制详见 [config §F1](config.md)。

### F8. 记忆的自然遗忘

记忆像人类记忆一样随时间自然衰退——长期未被引用的事件自动淘汰，频繁被引用的事件更加牢固。

- 每条事件创建后附带遗忘倒计时，倒计时耗尽后事件自动删除
- 不再关联任何事件的实体（0 引用）一并删除
- 已有事件在新会话挖掘中被再次识别时，遗忘倒计时大幅延长——表明该事件在多个会话中持续出现，值得长期保留
- 事件在会话中被搜索并注入时，遗忘倒计时小幅延长——表明该事件在当前仍有参考价值

## 非功能需求

- 搜索执行不阻塞主对话流程——搜索超时后主对话继续，不等待
- 规则文件的内容在所有 Agent 间一致可见——同一份规则文件被所有会话引用
- 记忆数据按 Agent 隔离，不同 Agent 的记忆互不干扰
- 配置修改后即时生效，无需重启服务
