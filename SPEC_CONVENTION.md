# SPEC 规格书编写规范

> 状态：**草稿 / 待完善**，本规范本身会随实践持续更新

---

## 什么是 SPEC

SPEC（规格书）是模块的**精确功能说明书**，描述"这个模块做什么、做到什么程度、接口是什么、边界在哪里"。

SPEC 不是：
- 需求文档（需求文档描述"要做什么"，SPEC 描述"系统现在是什么"）
- 开发计划（不含 Step 1/2/3，不含验收标准）
- 变更记录（不含 issue 号、PR 号）

SPEC 是代码的**镜像**，代码改了 SPEC 必须同步更新，SPEC 改了代码必须同步实现。

---

## SPEC 存放位置

```
src/<模块名>/SPEC.md
```

每个模块的规格书放在对应代码模块目录下，而不是统一放在 `docs/`。

例外：`docs/` 目录保留给：
- 项目级文档（README、贡献指南）
- 面向用户的指南、快速上手
- 与特定代码模块无强绑定的横向设计说明

---

## SPEC 文件命名

- 文件名：`SPEC.md`（每个模块根目录只有一个）
- 不使用 issue 号、日期等临时性标识
- 如需分文档描述子模块，在 `src/<模块>/docs/` 下组织

---

## SPEC 内容规范

### 必须包含

1. **模块职责**：一句话描述模块负责什么
2. **对外接口**：公开函数、数据结构、消息格式（精确描述，不含"待实现"）
3. **模块边界**：与哪些模块有依赖，边界在哪里
4. **核心数据结构**：关键 struct/enum 的字段说明
5. **行为规范**：系统在该模块范围内的客观行为描述（用"系统会..."、"模块负责..."，不用"应该..."、"将要..."）

### 禁止包含

- 开发步骤（Step 1/2/3）
- 验收检查项
- issue 号、PR 号
- 需求来源链接
- 工期估算
- 未来计划（"考虑支持..."、"后续可扩展..."）

### 语气规范

- 客观陈述：描述系统**现在是什么**，不是"系统应该是什么"
- 用肯定句，不用"如果...的话"、"可能"、"考虑"
- 接口描述必须精确，包含参数类型、返回值、错误处理

---

## 编写周期

- **需求阶段**：需求文档 → spec 是单向的，spec 还没写不代表不能开发
- **开发阶段**：developer 依据需求文档先写 SPEC，SPEC review 通过后再写代码
- **Review**：braino 对照需求文档检查 SPEC 准确性，发现偏差提 issue
- **上线后**：代码变更必须同步更新 SPEC，保持镜像一致

---

## SPEC 与需求文档的关系

- 需求文档：描述要做什么
- SPEC：描述系统现在是什么
- 开发前：需求文档 → SPEC（由 developer 完成，braino review）
- 开发后：代码变更 → SPEC 更新（由 developer 完成）
